CS-001 — ITP Referral Portal

Receives structured referrals from signed ITP partners and routes them to the CP1 claim intake queue. Pilot-scope build: sufficient to run the PartP1 T4 test, not the full production platform.

Date: 23 May 2026
Slice: 1 — ITP Partnership Loop
Flow: PartP1
Performance bound: PB-06 (referral ≤5 min)
Gate: TRR · Slice 1 exit
Tickets: 6
Pilot scope

Lightweight web form + email notification + manual log. No CRM integration, no automated routing, no claimant-facing output. The acceptance criteria for PartP1 (≥10 referrals, ≤5 min completion, NTP fee correctly attributed) can be met at this fidelity. The production build is commissioned after Slice 1 gate clears.

Process — what the component does

  1. ITP navigates to their unique referral URL (e.g. calmly.io/refer/[itp-token]). URL encodes ITP identity — no login required at pilot stage.
  2. ITP completes and submits form. Client-side validation prevents submission with missing required fields. No server round-trips mid-form.
  3. On submit: referral record written to database (timestamp, ITP token, all form fields). NTP fee record written (£25 against ITP account, pending monthly reconciliation).
  4. Intake specialist receives email notification: claimant name, harm type, claim value band, limitation period flag, ITP name. Link to referral record.
  5. Referral appears in intake specialist queue view (a simple list at pilot stage — no dashboard required). Status: Received.
  6. ITP receives submission confirmation on screen: "Referral received — Calmly will contact [claimant first name] within 2 working days." No email confirmation to ITP at pilot stage.

Outputs and interfaces

Output Destination Format at pilot stage

Acceptance criteria

These are the criteria against which the PartP1 T4 test is assessed. The portal must meet all of them before Slice 1 gate can be called.

  • ✓ AC-01
    Referral form is completable in ≤5 minutes from URL navigation to submission confirmation.
    Source: PB-06 (performance bound). Tested by timing 10 submissions with a representative ITP practitioner.
  • ✓ AC-02
    Intake specialist receives notification within 5 minutes of submission.
    Source: PartP1 pilot design. Measured from submission timestamp to email delivery timestamp.
  • ✓ AC-03
    NTP fee of £25 is correctly attributed to the referring ITP for each submission — verifiable by matching referral ID to fee log row.
    Source: ARM SC model. Verified by reconciling fee log against referral records at end of pilot window.
  • ✓ AC-04
    Claimant consent record is written for every submission. No CP1 intake begins without it.
    Source: legal requirement. Enforced by CP1 check: intake specialist cannot open an intake without a linked consent record.
  • ✓ AC-05
    ≥10 referrals received within the pilot window without active prompting from Calmly (the ITP refers without being chased).
    Source: PartP1 T4 acceptance criteria. This is the architectural test — it validates that the ITP has embedded referral as standard procedure, not one-off behaviour.
  • ✓ AC-06
    Unique ITP URL correctly identifies the ITP on every submission — no submissions are attributed to the wrong ITP or to an unknown ITP.
    Source: NTP fee integrity. Verified by inspecting ITP token on all referral records.

Out of scope for this build

  • ✗
    Automated routing to CP1 Email notification to intake specialist is sufficient at pilot stage. Automated queue management is a production build item, commissioned after Slice 1 gate clears.
  • ✗
    Claimant-facing confirmation email The ITP manages claimant communication at point of referral. Calmly contacts the claimant directly as part of CP1. No email sent to claimant from this component.
  • ✗
    ITP login / authenticated session Token URL is sufficient access control at pilot scale (≤25 ITP contacts). Authenticated ITP portal is a later build.
  • ✗
    CRM integration Referral records are stored in a simple table. No CRM sync, no Salesforce, no HubSpot. Pilot volume (target: 25–100 referrals) is manageable manually.
  • ✗
    Real-time NTP fee payment to ITP Monthly reconciliation is the ARM model at Stage 1. The fee log enables this — the payment mechanism itself is part of the Pool Operator infrastructure (Slice 4).
  • ✗
    OP5 outcomes reporting dashboard OP5 (ITP outcomes reporting) is a separate T3 component, built in parallel. This portal only writes the fee record and referral record that OP5 reads later.
  • ✗
    Document upload The claimant uploads documents during CP1 intake — not at referral. The ITP is not expected to handle claimant evidence.

Tickets

Six tickets. Build order: T1 → T2 → T3 and T4 in parallel → T5 → T6. T6 is the integration test — do not run until T1–T5 are complete.

ID Title Acceptance criteria Size
T001
Referral form — fields, validation, and submit
Build the web form with all 10 fields (9 ITP-entered + 1 auto from URL token). Client-side validation on required fields. Submit button disabled until required fields are complete. On submit: POST to /api/referrals. On success: show confirmation message.
AC-01: completable ≤5 min
Dropdown options match spec. Required fields enforce on submit. Confirmation visible on screen after submit.
 M
T002
ITP URL token — generation and read
Each ITP gets a unique URL containing an encoded token (e.g. base64 ITP ID). On page load, the token is parsed from the URL and stored in a hidden form field. If the token is missing or invalid: show an error page ("this referral link is not valid — contact Calmly"). Token must survive form submission and be included in the POST body.
AC-06: ITP correctly identified on every submission
Invalid or missing token prevents form from loading. Valid token passes through to referral record.
 S
T003
Referral record — database schema and write
Create referrals table. Columns: id, created_at, itp_token, itp_practitioner_name (nullable), claimant_name, claimant_email, claimant_phone (nullable), harm_type, claim_value_band, harm_timing, has_documents, consent_given (bool), consent_timestamp, status (default: 'received'). POST /api/referrals writes a row and returns the referral ID.
AC-04: consent record written on every submission
All required fields non-null after write. Status defaults to 'received'. Referral ID returned in response.
 S
T004
NTP fee log — write on referral submission
Create ntp_fee_log table. Columns: id, referral_id (FK), itp_token, amount (fixed £25.00), created_at, status (default: 'pending'). Write a row when a referral record is successfully created. This is the only financial record at pilot stage — no payment processing in scope.
AC-03: £25 attributed to correct ITP for every referral
One fee row per referral. amount always £25. itp_token matches referral record. status = 'pending'.
 S
T005
Intake notification — email on submission
On successful referral write: send email to intake specialist inbox. Subject: "New referral — [harm_type] · [claim_value_band] · [ITP name]". Body: claimant name, harm type, claim value band, harm timing, limitation flag (auto-set if harm_timing = '6+ years'), has_documents, ITP name, referral ID, link to referral record in queue view. Use transactional email provider (e.g. Resend or Postmark).
AC-02: notification delivered ≤5 min after submission
Email delivered to intake inbox. Limitation flag present when harm_timing = '6+ years'. Referral ID in email matches DB record.
 S
T006
End-to-end integration test
Using a test ITP URL, submit a referral and verify: (1) referral record written correctly, (2) fee log row written correctly, (3) intake email delivered within 5 min, (4) consent record is present. Then time 5 submissions by a non-technical person unfamiliar with the form — all must complete in ≤5 min. Document results. This is the acceptance test for AC-01 through AC-06.
All 6 ACs pass. Timing results documented. Any field or timing failure is a bug before Slice 1 pilot runs.
 M

Confirm tech stack choice (framework, DB, email provider). The spec is stack-agnostic — the only constraint is that the form must be web-accessible via URL (no app install required for the ITP) and the DB must support the referral and fee log schemas above.
