Governing thought

Architecture Definition tells you the structure holds. Design Definition tells you how to build each part of it. The gap between them is not a gap in ambition — it is a gap in precision. A verified BFF says the logic is sound. A CDR package says every flow is specified with enough rigour that someone could build it, test it, and hand it back for verification. Without Design Definition, implementation is improvisation on a sound blueprint.

Section 1

What Design Definition is

The INCOSE TP-5 purpose, adapted for IVE: Design Definition provides sufficient detailed data about each system element to enable implementation consistent with the architectural entities defined in Architecture Definition (TP-4). It does not redesign the architecture. It does not build anything. It specifies — with enough precision that each flow can be built, tested, and verified independently.

After the Design Loop produces a verified architecture (BFF with FMOS ≥ 60%, BALM Verification PASS, Consistency Audit PASS), you know the structure holds. You do not yet know how to build each flow within that structure. Design Definition closes that gap.

The gap it fills

The IVE Design Loop produces a verified BFF — a structural logic that holds under scrutiny. The PDR gate confirms FMOS ≥ 60%, BALM Verification PASS, and Consistency Audit PASS. What it does not produce is a build plan. The BFF tells you what the system must do and how its elements relate. It does not tell you how each flow is constructed, what its interfaces look like, or which flows should be built before which. Design Definition produces those outputs. CDR (Critical Design Review) is the gate that confirms they are complete.

What it is

The process of decomposing the verified architecture (BFF) into flow specifications detailed enough that each element can be built, tested, and verified independently. The output is a CDR package — flow register, flow specifications, interface control documents, requirements traceability matrix, and build sequence.

What it isn't

  • More architecture work. That is TP-4. The BFF is verified. Design Definition does not redesign it — if the BFF needs redesign, return to the Design Loop.
  • Building anything. That is TP-7. Specifications precede builds. No flow should be built before its spec is written and reviewed.
  • A product requirements document. A PRD captures what the customer wants. A flow specification captures how that flow satisfies an architectural requirement. They answer different questions.
⚠ CDR gate — most common failure mode

Specifications that describe what a flow is rather than what it must do. "Provides EV assessments" is a description. "Calculates the expected value of a claim within ±15% of expert median in ≤5 working days" is a specification. If an acceptance criterion cannot be turned into a test, the spec is incomplete.

A secondary failure: build sequences ordered by dependency but not by risk — starting with T2 flows and leaving T4 bets until the end. Architectural failures discovered late are expensive. Discovered early, they are signals.

Section 3

Flow taxonomy

Design Definition specifies flows, not static objects. A flow implies movement — product moving from the venture to the customer through a series of handoffs. Flows have natural properties: source, destination, carrier, trigger, rate, cost. The taxonomy organises flows by cut (which model they originate from), flow type, carrier, requirement types, and sources.

A flow delivered by a relationship manager has a completely different specification to the same flow delivered by an app. The Carrier field, drawn directly from the ARM, is what determines the design specification. It is not optional.

Carrier types

Role (PVC)

Direct personnel

Scales with volume. ARM source: PVC cost layer. Spec must work within that role's assumed capability and time budget. Training materials and SOPs are part of the flow design — they are not assumed to exist. The ARM's time budget for the role sets the upper bound on flow complexity.

Role (SC)

Indirect/support personnel

Semi-fixed overhead. ARM source: SC cost layer. As above, but at Territory or HQ coordination layer. Volume elasticity is lower. Spec must account for the fact that SC roles serve multiple LMUs simultaneously — single-LMU assumptions fail at scale.

System

Technology

ARM source: RC (variable) or IC (fixed). Spec becomes a software or platform requirement. Reliability and latency requirements apply. No human variability — the spec must be complete enough to test programmatically. Failure modes are deterministic.

Third-party

Contracted service

ARM source: RC (contracted). Spec becomes an SLA. Design controls what can be contractually required, not how it is executed. Failure modes are contractual — the spec must define what constitutes a breach and what the fallback is. No design control over execution.

1

Build the flow register

Extract the complete list of system elements from BFF, AOM, and CTM together. This is the flat inventory that all Design Definition work proceeds from.

IVE system elements are of two types. Product flows (from CTM) are the customer-facing flows that drive state changes: Working Product, Communications Product, Payment Product, and Partner Product. Operational flows (from AOM) are the internal flows that produce and deliver the product: LMU workflows across the Making, Selling, and Using streams; the Territory coordination layer; and the HQ platform.

Completeness test: every cost line in the AOM and every product flow in the CTM should appear in the register. If a cost line exists with no register entry, there is a gap. If a register entry has no AOM or CTM source, it is scope creep.

Worked specification — WP2: Independent EV assessment system

Flow name WP2 — Independent EV assessment system
Flow type Working Product flow — the core analytical engine. Produces the independent expected value report that is the mechanism for both C2 (disrupt defendant motivated reasoning) and C3 (shorten WC recovery cycle by eliminating defendants' source-credibility objection). All product value and all revenue traceability run through this flow.
Source → Destination Source: WP1 (eligibility assessment workflow) — passes a structured claim file to WP2 on PASS verdict.

Destination: CP2 (EV disclosure and claim activation) — receives the finalised EV figure (£ low/mid/high range) to disclose to the claimant and defendant simultaneously. WP2 also feeds ICD-04 to CP2 and logs each assessment to the C10 track record dataset (OP2).
Carrier (from ARM) Primary carrier — Third-party (RC): independent assessment organisation. University law clinic cohort or equivalent student-staffed body. Contracted at a per-assessment flat fee of ≤£16.28/claim (ARM validated; C3 SR1). Structurally independent of Calmly — no financial stake in any claim outcome. This independence is the operative mechanism for both C2 (defendant cannot apply motivated-source discount) and C6 (buyout operator can underwrite against the report). Independence contamination is a double failure mode: C2 and C6 fail simultaneously.

Secondary carrier — Role (PVC) QA: EV specialist. ARM role: EV assessment specialist at LMU. Per-claim time budget: ≤20 minutes quality review. ARM-derived headcount: 600 FTE across 50 LMUs at full capacity (350,000 claims/year). Dominant cost driver: EV specialist accounts for £100.29/claim (38.9% of total cost floor at full capacity). C9 QA framework governs volume allocation across assessment org pool — each claim routed to highest-quality available org based on accuracy, turnaround, and dispute rate metrics.
BALM source C1 — Circumvent the cost bottleneck (SR2, SR4, SR5): WP2 is the operational embodiment of the Workaround Strategy. The pooled, independent EV assessment creates the credible litigation backstop without a solicitor. Cost floor: £257.63/claim at scale (4.1× below conventional BFF midpoint of £1,050). Assessment org payment ≤£16.28/claim is the per-unit cost expression of C1 SR4.

C2 — Eliminate the biggest monetisable cost (SR4, SR5): WP2 produces the quantitative EV report that collapses both parties' self-serving assessments simultaneously. Report contains: (a) expected settlement value as a single reference-class-based figure; (b) escalation schedule showing what the settlement ask becomes at each stage of legal escalation. The accuracy of the EV figure is the direct driver of defendant settlement rate — and therefore of KMC realisation (claimant KMC: £1,233; defendant KMC: £1,400; total: £2,633/claim).

C3 — Circumvent the WC bottleneck (SR3, SR4): independence of the assessment source is the mechanism that eliminates defendants' tactical basis for delay within mediation. Settlement velocity (45-day vs 365-day recovery cycle) is the direct function of whether defendants can credibly contest the report on source-credibility grounds. WP2 source architecture (structural independence) is the C3 scaling strategy.

C9 — Resource moat (structural): WP2 feeds the track record dataset. Accuracy improves with volume. The methodology is owned by Calmly — portable across assessment orgs. Concentration limit: ≤30% per org (C9 structural competition constraint).
Function Produces an independent expected value assessment for each eligible claim — stated as a £ figure (low/mid/high range) with a disclosed, scrutinisable methodology — within ≤5 working days at a per-claim cost of ≤£16.28 to Calmly, such that ≥80% of assessments fall within ±20% of an expert-median reference set (barrister-calibrated) across a blinded validation sample of ≥30 claims.

Four sub-functions:
1. Route: assign each WP1-passed claim to the highest-available-quality assessment org (C9-governed allocation).
2. Receive: accept structured EV report within 5 working days; trigger quality review.
3. Review: EV specialist validates report against published methodology (≤20 minutes/claim); approves or returns for revision (≤1 revision cycle within 48 hours).
4. Record: log EV prediction, assessment org ID, dispute category, ITP source, and methodology version to C10 track record dataset for ongoing model calibration.
Behaviour Scenario 1 — Eligible claim received (standard path):
Trigger: WP1 returns PASS verdict + structured claim file.
Processing: EV specialist routes claim to highest-quality available org based on C9 QA metrics (accuracy score, rolling dispute rate, turnaround SLA compliance). Claim file transmitted to org via exchange platform with methodology version ID and turnaround deadline.
Org produces EV report: expected settlement value (£ low/mid/high), methodology narrative, evidence weighting rationale, confidence rating.
EV specialist receives report; ≤20-minute quality review against published methodology checklist.
Output: PASS → mark assessment PASS in exchange platform; ICD-04 fires to CP2 with finalised EV figure, claim ID, assessment org ID, timestamp, methodology version, quality caveats; prediction logged to C10 dataset.

Scenario 2 — QA review fail (revision cycle):
Trigger: EV specialist marks assessment FAIL in quality review.
Processing: report returned to assessment org with specific methodology failure annotation. Org has 48 hours to revise and resubmit.
Output: revised report received within 48 hours → re-enters quality review. If revision exceeds 48 hours or second review fails → escalate to HQ EV Methodology team (Head of EV Methodology, ARM role). Claimant informed of delay at 72 hours of total WP2 processing time.

Scenario 3 — Assessment org at or near capacity:
Trigger: primary assigned org reports full capacity (current queue + new claim exceeds SLA-safe load).
Processing: C9 QA framework reroutes to next-highest-quality available org in pool. ≥3 active orgs required at all times (C9 structural competition constraint). If fewer than 3 active orgs available → HQ pool operator relations team alerted within 24 hours to activate pipeline org.
Output: claim assigned to secondary org; turnaround SLA restarts from new assignment time.
Interfaces Receives from WP1: eligibility PASS signal + structured claim file (claimant details, dispute category, evidence package, ITP source ID). Trigger: WP1 PASS verdict. Latency: WP2 routing within 4 working hours of WP1 PASS (per ARM time budget).

Sends to CP2 via ICD-04: finalised EV figure (£ low/mid/high range), claim ID, assessment org ID, assessment timestamp, methodology version, quality review caveats. Trigger: EV specialist marks assessment PASS. Latency: CP2 session scheduled within 48 hours of WP2 PASS.

Sends to OP4 (C9 QA framework): assessment report + quality review score + dispute category (if revision required) + turnaround time. Used by C9 framework to update org accuracy scores and volume allocation weights.

Sends to OP2 (C10 data infrastructure): EV prediction (£ mid figure), claim category, ITP source, assessment org ID, methodology version, outcome (recorded at settlement/judgment — retroactive feed). Builds the accuracy-over-time dataset that is Calmly's data moat.
Constraints FMOS per-unit cost ceiling (ARM): per-claim payment to assessment org ≤£16.28. This is the ARM-validated figure from C3 SR1. At 350,000 claims/year, total assessment org expenditure = £5,698,000. Breach of this ceiling requires re-verification of C1 cost floor and FMOS recalculation.

Independence requirement (regulatory and architectural): assessment org must be structurally independent of Calmly and of the buyout operator. Assessors may hold no financial interest in any claim outcome — no performance-linked fee, no equity, no revenue share. This constraint is architectural: it is the mechanism that makes both C2 (motivated-reasoning disruption) and C6 (buyout underwriting) possible. Contamination of independence disables both simultaneously.

Concentration limit — ≤30% per org (C9): no single assessment organisation may handle more than 30% of WP2 volume in any rolling 90-day period. Governs volume allocation in all three scenarios. Enforced by C9 QA framework (OP4).

Assessment org minimum pool size (C9): ≥3 active orgs operational at all times. Prevents single-org dependency and preserves competitive accuracy pressure across the pool.

Without-prejudice framework (regulatory): EV figure and escalation schedule disclosed under without-prejudice cover — cannot be used as evidence of admission in other proceedings. This constraint applies to the report format and disclosure protocol (CP2 carries the disclosure; WP2 produces the figure that CP2 discloses).
Acceptance criterion All five conditions must be met to declare WP2 verified:

AC1 — Accuracy test (blinded comparison vs barrister reference set): across ≥30 blinded test claims assessed by the org pool, ≥80% of EV figures (mid-point) fall within ±20% of the expert median produced by an independent barrister reference panel assessing the same claims. Test design: claims selected to span all three dispute categories (housing, employment, consumer); assessors have no access to barrister verdicts before submission.

AC2 — Turnaround test: average org-to-delivery time ≤5 working days across the ≥30 test claims. No single claim exceeds 8 working days without triggering the revision/escalation protocol.

AC3 — QA review efficiency test: EV specialist completes quality review in ≤20 minutes per claim across the test set. Measured by exchange platform timestamp (claim received → PASS/FAIL marked).

AC4 — Dispute rate test: quality review failure rate (reports requiring revision) ≤15% across the test set.

AC5 — Cost test: total per-claim payment to assessment org ≤£16.28. The EV specialist QA time sits within the ARM-modelled LMU people cost (£100.29/claim at scale) and is not separately capped here. FAIL on AC5 triggers re-verification of C1 cost floor and FMOS recalculation.

FAIL on AC1 or AC5 returns the architecture to C3 design. FAIL on AC2–AC4 alone triggers process redesign within WP2 without architectural return.
Assumption tier T4 — three assumptions are untested and together drive the FMOS model's validity:

1. EV accuracy (AC1): the assumption that university law students operating under a published methodology can produce EV figures within ±20% of expert-median is the central T4 bet in the architecture. If this fails, C2's motivated-reasoning disruption mechanism fails and C3's recovery-cycle shortening mechanism fails simultaneously. FAIL consequence: architectural return to C3 SR3 — Scaling Strategy must be redesigned.

2. Turnaround time (AC2): the assumption that assessments can be produced in ≤5 working days is untested. Drives the working capital model (£2.2M peak WC at Y5 at 70% settlement rate). A 10-working-day average would roughly double WC requirements. FAIL consequence: revise WC model in FinSim; re-evaluate FMOS impact before architectural return.

3. Per-claim cost (AC5): the £16.28 ceiling is derived from the C3 working capital model and ARM. It has not been tested against a live assessment org. FAIL consequence: C1 cost floor rises; re-verify FMOS before architectural return.

Validation protocol: structured pilot of 20–30 live assessments with a real assessment org cohort (law clinic, LPC programme, or equivalent) before any exchange platform build commits to WP2 as designed. Build highest-T4 flows first per Step 5 risk-ordering principle.

Worked specification — CP1: Claim intake and eligibility disclosure

CP1 is the claimant-facing intake session. It begins when an ITP-referred claimant receives an outreach from the LMU intake specialist, and ends when the claimant either (a) submits a complete, evidence-supported claim record — triggering WP1 — or (b) is disqualified and receives a no-proceed notification. The flow carries BALM C4 (pool loss insurance eliminates claimant doubt about financial survivability of the pool) and C5 (Coase assignment eliminates anticipated litigation burden). Both challenges are partially addressed in CP1: C5 by bounding the claimant's active role to this single guided session; C4 by communicating the insurance backstop during the session.

Flow name CP1 — Claim intake and eligibility disclosure
Flow type Communications Product — a structured sequence of interactions between the LMU intake specialist and the claimant that transforms an ITP referral into a verified, evidence-supported claim record or a confirmed non-proceed.
Source → Destination ITP referral (Part1, via ICD-01) → Eligibility assessment workflow (WP1, via ICD-02)

On non-proceed: CP1 → no-proceed notification (claimant only; no WP1 handoff).
Carrier Role (PVC) — LMU intake specialist. At-scale headcount: 350 FTE across 50 LMUs (7 per LMU). Role split within the 69-minute budget is functional, not structural.
BALM source C4 — Circumvent customer doubt (claimant activation). Key Efficacy Doubt: pool financial survivability. CP1 addresses this by communicating pool loss insurance as a standard feature during the intake session — the claimant's first point of contact after ITP referral. The insurance framing converts an abstract promise ("your claim is pooled with others") into a concrete guarantee ("if the pool falls short, insurance covers the gap"). C4 Olson (1965) coordination failure + Arrow (1963) insurance logic.

C5 — Eliminate learning disruption. Primary block: anticipated litigation complexity (what happens after I submit?). CP1 directly addresses this by confining the claimant's entire active role to this one guided session — the specialist handles all subsequent steps. The session script must explicitly communicate Coase assignment (Coase 1937 + Williamson 1985) in plain language: "Once you submit, we handle everything. You do not attend hearings. You do not instruct lawyers." Secondary block: evidence assembly anxiety (step 2). CP1 addresses this via guided submission with real-time specialist support. Bandura (1986) self-efficacy framework.
Function CP1 transforms an ITP-sourced referral into a verified, evidence-supported claim record by:

(a) Pre-screening the referred claimant against eligibility criteria (dispute category, claim value floor, limitation period, prior legal action) to determine whether a full submission is warranted;
(b) Guiding the claimant through a structured submission session (≤8 screens, ≤5 mandatory fields per RTM C4) to capture claim details and evidence;
(c) Communicating pool loss insurance and Coase assignment during the session to eliminate C4 doubt and C5 learning disruption;
(d) Handing off a complete claim record to WP1 on pre-screen pass, or issuing a no-proceed notification to the claimant on pre-screen fail.
Behaviour CP1 executes in two sub-phases:

Sub-phase 1 — Pre-screen call (outbound, ~15 min): intake specialist contacts claimant within 2 working days of ITP referral receipt (per ICD-01 latency). Applies four binary eligibility gates: (G1) dispute category in scope; (G2) claim value ≥ floor (TBD at pilot, indicative £200); (G3) limitation period live (≤6 years for contract, ≤3 years for most tort); (G4) no prior court judgment on the same dispute. FAIL on any gate → no-proceed notification to claimant; record logged as ineligible. PASS all four gates → proceed to sub-phase 2.

Sub-phase 2 — Guided submission session (~30–40 min, online or phone-assisted): specialist guides claimant through intake form. Mandatory fields: (F1) full name and contact details; (F2) defendant name and contact details; (F3) dispute category (pre-populated from G1); (F4) incident date and brief description (≤500 chars); (F5) evidence file upload (at least one document: invoice, contract, correspondence). Optional fields: (F6) prior resolution attempts (letter before action Y/N); (F7) preferred contact method. During session, specialist delivers the C4/C5 script (pool insurance + passive role explanation). Session ends with claimant confirmation and digital signature. Record transmitted to WP1 via ICD-02.

Timing: pre-screen call within 2 working days of ITP referral; submission session within 5 working days of pre-screen pass; total CP1 elapsed time ≤7 working days from ITP referral receipt.

No-proceed notifications are sent within 1 working day of gate fail, with brief reason (category, value, limitation, or prior judgment) and a signpost to Citizens Advice or MCOL for out-of-scope claims.
Interfaces ICD-01 (inbound): Part1 → CP1. Already specified. Data: claimant name, contact details, ITP source ID, dispute category, brief description (≤500 chars), ITP staff member ID. Trigger: ITP staff submits form. Latency: record within 2 min, intake assessor notified within 15 min, CP1 session within 2 working days.

Trigger: claimant completes digital signature in sub-phase 2. Latency: record available to WP1 eligibility assessor within 15 minutes of signature.

Constraints C1 — ARM time budget: 69 min/confirmed claim for all intake specialist activities. Sub-phase 1 target: ≤15 min. Sub-phase 2 target: ≤40 min (specialist active time, not elapsed session time). Remaining 14 min: admin, record-keeping, no-proceed notifications. Budget is tight; scripts must be pre-prepared and specialist must not free-form the C4/C5 explanation.

Worked specification — WP1: Eligibility assessment workflow

WP1 is the internal eligibility adjudication step. It sits between CP1 (which produces a complete claim record) and WP2 (which produces the expected-value assessment). WP1's function is binary: confirm that the claim record produced in CP1 meets the substantive criteria for pool admission, and pass it to WP2 — or reject and close. The FMOS impact of WP1 is structural: every claim rejected here avoids £150+ in EV assessment costs (WP2), making WP1 the primary cost-control gate in the architecture.

Flow name WP1 — Eligibility assessment workflow
Flow type Working Product — an internal operational process carried out by the LMU eligibility assessor (same role as intake specialist at LMU level) that produces a binary PASS/FAIL eligibility determination for each submitted claim record.
Source → Destination Claim intake and eligibility disclosure (CP1, via ICD-02) → EV assessment workflow (WP2, via ICD-03 — to be specified)

On FAIL: WP1 → no-proceed notification to claimant; claim record archived. No WP2 handoff.
Carrier Role (PVC) — LMU eligibility assessor. At-scale: functionally the same role as CP1 intake specialist (AOM "intake assessors (350)"; C1 cost floor model does not separate the two roles). ARM time budget: the 69-min/confirmed-claim intake specialist allocation spans both CP1 facilitation and WP1 adjudication. WP1-specific adjudication time is estimated at 15–20 min per submitted claim (within the 69-min budget), on the basis that: (a) the pre-screen in CP1 already eliminates clear out-of-scope claims; (b) WP1 adjudication applies a structured checklist rather than a discretionary legal judgment; (c) FOS caseworker analogues at ~97 resolved cases/FTE/year (FOS 2024-25: 2,344 FTE, 227,445 resolutions) imply complex adjudication runs far slower than a binary eligibility check — WP1 is structurally simpler. Salary proxy: as CP1 carrier, blended ~£20/hr.
BALM source C1 — Cost floor (eligibility screening prevents wasted EV assessments). WP1 is the primary cost-control mechanism in the C1 model. An unscreened claim reaching WP2 costs ~£150+ in EV assessment (assessment org payment + EV specialist QA time). At a 35% post-CP1 ineligibility rate (T4: 65% pass), WP1 prevents £150 × 0.35n in wasted EV spend. At 1,500 claims/year (AOM v2 equilibrium), this prevents ~£78,750/year in avoidable EV cost. The FMOS calculation (63.2%) assumes WP1 operates as specified — if ineligible claims leak through to WP2, the cost floor rises and FMOS falls. C1 Schelling (1960) precommitment logic: WP1 is the mechanism that makes the pool cost-credible.

C4 — Circumvent customer doubt. WP1 also directly supports C4 by ensuring the pool contains only admitted, verified claims. Pool loss insurance pricing (C4 mechanism) depends on a predictable claim type distribution. Ineligible claims admitted to the pool would distort that distribution and risk repricing or withdrawal of pool insurance. WP1 quality therefore underwrites the C4 mechanism's financial stability.
Function WP1 determines pool admission eligibility for each submitted claim record by applying a structured adjudication checklist to the claim record received via ICD-02. The function produces either:

(a) PASS — the claim meets all eligibility criteria; claim record is transmitted to WP2 via ICD-03 for expected value assessment; or
(b) FAIL — the claim does not meet one or more eligibility criteria; claimant receives a no-proceed notification; claim is archived with reason code.

WP1 does not produce an expected value, express a view on claim merits, or make any legal determination. It applies administrative eligibility criteria only.
Behaviour WP1 executes as a structured checklist review of the ICD-02 claim record. The eligibility assessor applies six adjudication criteria in sequence:

E1 — Evidence completeness: at least one supporting document uploaded (invoice, contract, correspondence). No document → FAIL.
E2 — Dispute category confirmed in scope: category field matches the published scope list (housing, employment, consumer goods/services — pilot categories). Out-of-scope → FAIL.
E3 — Claim value in range: stated claim value ≥ floor (indicative £200) and ≤ ceiling (indicative £25,000 — above ceiling, claim is referred to specialist legal pathway, not pooled). Outside range → FAIL with appropriate referral signpost.
E4 — Limitation period: incident date confirms limitation is live. Expired limitation → FAIL (notified with expiry date and reason).
E5 — No prior judgment: claimant confirms (checkbox in CP1) no prior court judgment on the same dispute. Assessor spot-checks via MCOL public register for claims above £500. Prior judgment found → FAIL.
E6 — No active legal representation: claimant is not currently represented by a solicitor on this matter. Represented claimants are out of scope for pool admission (conflict with C5 Coase assignment — pool operator cannot assume management if solicitor is already engaged). Currently represented → FAIL with referral to their solicitor.

All six criteria PASS → record transmitted to WP2 via ICD-03 with eligibility PASS status, assessor ID, and timestamp.

Timing: WP1 adjudication target ≤1 working day from receipt of ICD-02 claim record. At 15–20 min per claim, an assessor can complete 3–5 WP1 adjudications per working day alongside CP1 facilitation duties (within 69-min/claim ARM budget at the LMU claim throughput rate of ~480 claims/module/year at AOM v2 equilibrium).
Interfaces ICD-02 (inbound): CP1 → WP1. To be specified. Data: claim record (F1–F7 from CP1 sub-phase 2), evidence files, eligibility gate results (G1–G4 PASS), pre-screen assessor ID, session timestamp, C4/C5 script delivery confirmation. Trigger: claimant completes digital signature in CP1 sub-phase 2. Latency: record available to WP1 assessor within 15 minutes.

ICD-03 (outbound): WP1 → WP2. To be specified. Trigger: assessor marks all six criteria PASS in platform. Latency: record available to EV specialist within 15 minutes of PASS mark.

Trigger: assessor marks any criterion FAIL. Latency: notification sent within 1 working day of FAIL determination.
Constraints C1 — Adjudication must not constitute legal advice. The eligibility assessor applies administrative criteria against a published checklist. Any assessment of claim merits, liability, or probable outcome is out of scope and creates SRA regulatory risk. The six criteria (E1–E6) are intentionally administrative.

C2 — WP1 must be completed within 1 working day of ICD-02 receipt. Breach creates a dead-time gap in the claimant's experience (they have submitted but received no decision), which is a C5 risk — extended silence reactivates anticipation of complexity.

If monthly claim intake spikes, adjudication queues must not exceed 2 working days. Queue management protocol required before platform launch.

C4 — The E5 spot-check (MCOL register) applies only to claims ≥£500. Below this threshold, the checkbox self-declaration in CP1 is accepted without verification. This is a known data quality risk — mitigated by post-settlement audit (if claimant misrepresented prior judgment, pool operator has a fraud claim against claimant).
Acceptance criterion All three conditions must be met to declare WP1 verified:

AC1 — Adjudication accuracy: independent review of ≥30 WP1 adjudications (blinded re-adjudication by a second assessor) must find ≥90% concordance (same PASS/FAIL outcome). Concordance <90% indicates criteria ambiguity and requires criteria clarification before scale. Measured in structured pilot.

AC3 — Post-EV proceed rate (C4 cost test): ≥70% of claims reaching WP1 PASS proceed to complete pool admission (CP2 and PP-C). This is a downstream test (measured at PP-C, not at WP1 itself) but WP1 eligibility quality drives it — if WP1 admits marginal claims that claimants abandon at EV disclosure (because EV is unexpectedly low), WP1 criteria need tightening. Architecture T4 target: 80% post-EV proceed rate. FAIL threshold for architectural review: <60%.
Assumption tier T4 (moderate) — two assumptions drive FMOS sensitivity:

AC2 — Acceptance rate (PB-07): across ≥50 live claim routings per active operator, acceptance rate ≥90% per operator. Measured over first 90 days of live routing. FAIL: operator removed from active pool; pool relations team reviews acceptance criteria mismatch.

AC3 — Concentration compliance: no single operator exceeds 30% of routed claims in any 90-day window across the pilot period. If QA Tracking System enforcement is working, this should be automatic — AC3 is a system verification check, not just an outcome measure.

AC4 — Onboarding cycle time: new operator from first meeting to signed framework agreement in ≤60 days. Measured across ≥2 onboarding cycles in the pilot. FAIL: pipeline operator acquisition takes too long to maintain the ≥3 active operator buffer; onboarding process to be streamlined.

Flow specification — WP3: Pool matching algorithm

Derived from: C3 architecture (design memo v2, 18 May 2026), AOM v2, Capella model (ICD-05, ICD-06, PB-07), resourcing model (21 May 2026). Assumption tier T2 — algorithmic matching is a solved technical pattern; risk is operational (operator acceptance rate), not architectural.

Flow ID WP3
Flow name Pool matching algorithm
Flow type Working Product — the automated routing engine that transforms an activated claim record (ICD-05 from CP2) into a matched pool operator assignment (ICD-06 to CP3). WP3 is the operational mechanism by which C3's WC = £0 property becomes real: no claim reaches a defendant without first being matched to an operator who will bear the litigation working capital.
BALM sources C3 — Circumvent the WC bottleneck (SR3, SR4): WP3 is the direct expression of C3's structural claim. The algorithmic match assigns each activated claim to a pool operator before any defendant notification is dispatched. The pool operator, not Calmly, bears working capital from this point. Calmly's WC per claim = £0 is arithmetically true only if every ICD-05 event is followed by a successful ICD-06 match. A routing failure (no operator available, or operator rejects the match) breaks the WC = £0 property for that claim.

C6 — Defendant can only be notified if matched: CP3 (defendant notification) is gated on ICD-06. This sequencing is architecturally enforced: the exchange platform cannot dispatch defendant notification until WP3 has confirmed a match. This prevents Calmly from notifying defendants for claims it cannot yet route to a buyer — which would create a WC liability with no corresponding recovery mechanism.
Carrier System (RC): Exchange Platform (OP1). WP3 is a fully automated flow — no human carrier in the standard path. The exchange platform executes the matching algorithm on receipt of ICD-05 and fires ICD-06 without manual intervention. The HQ Pool Relations Manager (Part2) maintains the operator routing weights that govern algorithm behaviour; but the routing decision itself is automated. ARM does not allocate any headcount to WP3 directly — it is an embedded function of the exchange platform (OP1), whose platform engineering headcount (AOM: 3–5 FTE) is allocated at the OP1 level.
Function On receipt of an activated claim record via ICD-05, WP3 selects the highest-weighted available pool operator whose acceptance criteria match the claim's attributes (dispute category, EV range, defendant geography) and whose current 90-day volume share is below the C9 concentration ceiling. The match fires ICD-06 to CP3 within ≤2 minutes of ICD-05 receipt. WP3 also updates Part2's QA tracking with the routing event, triggering the PB-07 acceptance rate monitoring cycle.

Four sub-functions:
1. Receive: ingest ICD-05 (activated claim record: claim ID, EV figure, dispute category, claimant commitment timestamp).
2. Score: apply matching score to each active pool operator — inputs: operator quality score (Part2 QA data), current capacity (declared capacity minus current claim queue), 90-day volume share (C9 enforcement), and acceptance criteria overlap with claim attributes.
3. Route: assign claim to highest-scoring operator; generate ICD-06 (pool match event).
4. Log: record routing event, operator assigned, and timestamp to Part2 QA Tracking System; update operator volume share counter.
Behaviour Scenario 1 — Standard match (≥1 eligible operator available):
Trigger: ICD-05 received from CP2 (claimant has committed to pool admission).
Processing: algorithm scores all active operators against claim attributes and C9 constraints. Highest-scoring eligible operator selected. ICD-06 generated and transmitted to CP3 within ≤2 minutes. Routing event logged to Part2 QA system.
Output: CP3 receives ICD-06 and proceeds to defendant notification dispatch. Claim status updated to MATCHED in exchange platform.

Scenario 2 — No eligible operator (temporary capacity constraint):
Trigger: all active operators are at declared capacity or have hit the C9 30% ceiling for the current 90-day window.
Processing: claim enters a routing queue in the exchange platform (PENDING_MATCH status). Claimant receives an automated notification: "Your claim is queued for pool assignment — expected matching within [X] working days." HQ Pool Relations Manager alerted within 1 working hour if queue depth exceeds 5 claims or wait time exceeds 5 working days.
Output: claim routes on the next available capacity slot. No claim is dropped; no defendant is notified until ICD-06 fires.

Scenario 3 — Operator rejects matched claim (post-ICD-06):
Trigger: pool operator returns a rejection response to ICD-06 within their 24-hour acceptance window.
Processing: WP3 scores remaining operators and routes to the next-highest eligible operator. Rejection logged to Part2 QA system for the rejecting operator's PB-07 acceptance rate calculation. If no second eligible operator exists, claim enters the routing queue (Scenario 2).
Output: second ICD-06 generated to next operator. If ≥2 rejections occur within 48 hours, Part2 quality protocol escalates to HQ Pool Relations Manager for manual review of claim attributes against operator acceptance criteria.
Interfaces Receives via ICD-05 (inbound from CP2): activated claim record — claim ID, EV figure (£ low/mid/high), dispute category, claimant commitment timestamp, ITP source ID. Trigger: claimant confirms pool admission in CP2 session. Latency: ICD-05 available to WP3 within 2 minutes of claimant confirmation. (Source: Capella ICD-05 — "Activated claim record (claim ID, EV figure, claimant commitment, timestamp)".)

Sends via ICD-06 (outbound to CP3): pool match event — matched claim ID, pool operator ID, defendant contact details (from claim record), EV figure. Trigger: WP3 matching algorithm selects operator. Latency: ICD-06 fires within ≤2 minutes of ICD-05 receipt in the standard path. (Source: Capella ICD-06 — "Pool match event (matched claim ID, pool operator ID, defendant contact details, EV figure)".)

Sends to Part2 QA Tracking System: routing event log — claim ID, operator assigned, routing timestamp, operator score at time of routing. Used by Part2 to update operator performance metrics and volume share counters.
Constraints C6 sequencing gate: ICD-06 must fire before CP3 dispatches any defendant notification. This is a hard architectural constraint — enforcement is via platform logic, not process discipline. The exchange platform must not permit CP3 to proceed without a confirmed ICD-06 record for the relevant claim ID.

C9 concentration enforcement: WP3 must actively exclude operators who have reached 30% of 90-day volume from receiving further ICD-06 assignments until the window resets. This exclusion is a routing constraint, not an advisory — the algorithm must enforce it mechanically.

Routing latency (≤2 minutes): the standard-path match must complete within 2 minutes of ICD-05 receipt. This is derived from the AOM design intent that the defendant notification sequence begins within hours of claimant commitment — a routing delay of hours (rather than minutes) would introduce dead time visible to the claimant. T2 sourcing: matching algorithm design patterns in litigation finance and insurance claims routing achieve sub-minute routing at comparable claim volumes.

Minimum operator pool: WP3 requires ≥1 eligible active operator to function at all. If the Part2 condition (≥3 operators) degrades to 0 eligible operators, WP3 cannot route and all claims queue. This failure propagates to CP3 (defendant notification stalls) and to PP-C (no settlement proceeds, no fee revenue).
Assumption tier T2 — algorithmic matching of claims to operators by discrete attributes (category, value range, capacity) is a standard routing pattern, well-attested in litigation finance, insurance claims, and marketplace platform literature. The architectural risk in WP3 is not the algorithm itself but the operator-side behaviour it depends on (PB-07: ≥90% acceptance rate). That risk sits in Part2, not in WP3.

T2 sourcing — matching algorithm design patterns: Burford Capital and Harbour (litigation finance) route diversified claim portfolios algorithmically against declared investment criteria. Insurance claims platforms (e.g., Guidewire, Duck Creek) route claims to handlers by category and complexity in <60 seconds at scale. No published paper is cited here because the algorithmic claim routing pattern does not require theoretical grounding — it is an engineering implementation problem, not a design assumption. The T2 designation reflects the maturity of the routing pattern, not a specific publication.

Residual T4 (inherited from Part2): if pool operators' actual acceptance criteria are materially narrower than their declared criteria — e.g., they reject claim categories they agreed to accept — WP3's routing accuracy degrades and Scenario 3 (operator rejection) becomes the norm rather than the exception. This risk is owned by Part2's pilot test design (T4 condition: heads of terms with ≥1 operator before WP3 build is committed).
Acceptance criteria All four conditions must be met to declare WP3 verified:

AC1 — Routing latency: ≥95% of ICD-05 events result in ICD-06 within ≤2 minutes in the standard path (Scenario 1), measured over ≥100 live routings in pilot. FAIL consequence: platform engineering review of algorithm performance — not an architectural return.

AC2 — Routing success rate: ≥90% of ICD-05 events result in an accepted ICD-06 match within 24 hours (i.e., Scenario 2 and Scenario 3 combined affect <10% of routings). This is the PB-07 acceptance rate test applied to WP3's output. FAIL consequence: operator pool quality issue — escalate to Part2 review rather than WP3 redesign.

AC3 — C9 enforcement accuracy: zero routing events in which an operator exceeds 30% of 90-day volume share. Measured continuously via QA Tracking System audit. FAIL consequence: platform engineering defect — routing weight enforcement logic to be corrected before any further live claim routing.

AC4 — Queue clearance time: when Scenario 2 (routing queue) is triggered, the queue clears within 5 working days for ≥90% of queued claims. Measured over any queue event in the pilot. FAIL consequence: Part2 pipeline operator count insufficient; add operator before scaling claim volume.

Flow specification — PP-C: Payment flow — claimant fee collection

Derived from: R6 FIT Verifier (20 May 2026) — contingent fee contracting, Rubinfeld & Scotchmer (1993); design memo v2 (18 May 2026); FinSim v4 (21 May 2026) — fee rate sensitivity table; Capella model (ICD-07 to PP-D; PP-C function description). Assumption tier T3 — contingent deduction mechanism is analogous to UK DBA practice; specific fee rate is T4.

Flow ID PP-C
Flow name Payment flow — claimant fee collection
Flow type Payment Product — the mechanism by which Calmly deducts its claimant-side platform fee from settlement proceeds at the point of resolution, then transfers net recovery to the claimant. PP-C is the revenue realisation event for Calmly's claimant fee line: no payment precedes it, and no payment obligation survives it.
BALM sources C6 — No-win-no-fee architecture (claimant side): the claimant pays only at settlement — and only from settlement proceeds. There is no cash flow event in the claimant's experience before PP-C fires. This is not a deferred payment arrangement: it is an architecture where no payment is possible before value is delivered, because the fee is extracted from value delivered. Theory: Rubinfeld & Scotchmer (1993) on contingent fee contracting — when payment is deducted from recovery proceeds, the supplier bears litigation risk and the client bears zero liquidity cost; contingent payment eliminates client-side adverse selection and liquidity barriers simultaneously. Dana & Spier (1993) confirm contingent contracts are efficient when the client is liquidity-constrained. (Source: R6 FIT Verifier, 20 May 2026, Pass — pp. 3–4.)

Financial model driver (FinSim v4): PP-C is the primary revenue realisation point for Calmly's claimant fee line. At the central scenario (£3,000 average settlement, 25% fee rate), PP-C generates £750/claim in gross claimant fee revenue, yielding FMOS of 65.6% against the £258/claim cost floor. Fee rate sensitivity is significant: at £2,000 average settlement and 15% fee rate (£300/claim), FMOS falls to 14.0% — below the F1 gate. The fee rate and average settlement are the two variables PP-C makes concrete; both are T4 pending pilot data. (Source: FinSim v4 sensitivity table, 21 May 2026.)
Carrier System (RC): Exchange Platform (OP1). PP-C is a fully automated payment flow — triggered by an upstream settlement event and executed without human intervention. The exchange platform holds the settlement proceeds in a designated client account (FCA client money rules) from the point the defendant's D3 payment clears until the fee deduction and net transfer are complete. The Head of Finance (ARM: 0.5 FTE at Stage 1, 1.0 FTE from Stage 2) provides compliance oversight of the client account and fee disbursement reconciliation — but the payment event itself is system-automated.
Function On receipt of a settlement event from PP-D (the defendant fee collection flow — triggered when the defendant's D3 buyout payment clears), PP-C executes a two-step payment sequence:

1. Deduct: apply the Calmly claimant platform fee (a defined percentage of gross settlement proceeds) to the settlement amount. Record the deduction as a transaction event in the exchange platform with: claim ID, gross settlement amount, fee rate applied, fee amount, net recovery amount, timestamp.
2. Transfer: transfer the net recovery amount to the claimant's designated bank account (captured during CP1 sub-phase 2 session). Transfer must complete within 2 working days of the settlement event. Confirmation sent to claimant by platform notification.

PP-C does not set the fee rate — that is an architecture-level parameter defined by the financial model (FinSim v4) and disclosed to the claimant in writing during CP2. PP-C enforces the rate; it does not negotiate it.
Behaviour Standard path — settlement proceeds received:
Trigger: PP-D confirms D3 resolution event (defendant's payment clears into Calmly client account). Capella: ICD-07 carries the D3 resolution event from CP3 to PP-D; PP-C is triggered by the downstream settlement confirmation from PP-D — no named ICD in the canonical model for the PP-D → PP-C handoff (parallel payment flow).
Steps: (1) Exchange platform receives settlement confirmation from PP-D with: claim ID, gross settlement amount, defendant ID, legal closure reference. (2) Platform applies claimant fee rate to gross amount: fee amount = gross settlement × fee rate. (3) Net recovery = gross settlement − fee amount. (4) Net recovery transferred to claimant's bank account (from CP1 record). Target latency: ≤2 working days of settlement event. (5) Transaction record created: claim ID, gross settlement, fee rate, fee amount, net recovery, transfer timestamp, bank account reference. (6) Claimant confirmation notification sent: "Your settlement has been processed. Net recovery of £[X] has been transferred to your account. Reference: [claim ID]."

Exception path — bank account details invalid or account closed:
Trigger: transfer fails due to invalid account details (as returned by payment processor).
Steps: (1) Platform flags claim ID as TRANSFER_PENDING. (2) Claimant contacted within 1 working day to provide updated bank details. (3) Transfer re-attempted within 1 working day of updated details received. (4) If claimant uncontactable within 10 working days, matter escalated to Head of Finance; funds held in client account pending resolution (FCA client money rules apply).
Interfaces Receives settlement event from PP-D (inbound, parallel payment flow): gross settlement amount, claim ID, defendant ID, D3 legal closure reference. No named ICD in the Capella canonical model for this handoff — PP-C and PP-D are parallel payment flows both triggered by the D3 resolution event, with PP-D handling the defendant fee and PP-C handling the claimant fee from the same settlement proceeds. The exchange platform holds the consolidated settlement proceeds until both flows have completed their deductions.

Reads from CP1 claim record: claimant bank account details (F1 — captured during intake). PP-C does not re-request bank details at the point of settlement — they are stored in the exchange platform from CP1 and validated at that point. If details have changed, the exception path (above) applies.

Sends to OP2 (C10 data infrastructure): resolved claim record — claim ID, gross settlement, fee amount, net recovery, category, ITP source, timestamp. Used for FMOS tracking, fee rate validation against FinSim v4 assumptions, and the C10 track record dataset.
Constraints FCA client money rules: settlement proceeds received from the defendant must be held in a designated client account (FCA CASS rules) from receipt until disbursement. Calmly's fee deduction and net transfer must both be executed from client account — no commingling with Calmly operating funds. The Head of Finance (ARM role) is the designated CASS oversight owner. This constraint governs the account structure of the exchange platform's payment rail (OP1).

Fee rate — architecture parameter, not PP-C variable: the fee rate applied by PP-C is set at the architecture level and disclosed to the claimant in CP2. PP-C enforces the disclosed rate; it cannot vary per claim or per claimant. Any change to the fee rate requires a CP2 disclosure update and re-validation of FinSim v4 FMOS calculations.

No-win-no-fee sequencing: PP-C may only fire after a confirmed settlement event from PP-D. It must not trigger on an unconfirmed or conditional settlement — e.g., a signed buyout agreement that has not yet cleared payment. The D3 event definition (defendant payment cleared, legal closure reference issued) is the trigger; the agreement alone is not sufficient.

Transfer latency (≤2 working days): net recovery must be in the claimant's account within 2 working days of settlement event. This is an FCA consumer outcome expectation (Consumer Duty — acting to deliver good outcomes for retail customers); it is not an internal SLA only.
Assumption tier T3 for the mechanism; T4 for the fee rate.

T3 sourcing — contingent fee deduction (mechanism): the deduction of a percentage fee from settlement proceeds is the standard practice under UK Damages-Based Agreement (DBA) regulations (The Damages-Based Agreements Regulations 2013, SI 2013/609 — legislation.gov.uk). Under DBA rules, a representative may deduct a percentage of damages (capped at 50% for most civil claims, 25% for personal injury, 35% for employment) from the client's recovery at the point of settlement. Calmly's PP-C mechanism is structurally identical: fee deducted from proceeds at resolution, net recovery transferred to claimant, no upfront charge. The DBA regulatory framework confirms the mechanism is legally operative and consumer-tested in the UK. Calmly is not a law firm and will not operate under DBA directly — but the mechanism is directly analogous, and the FCA authorisation framework (exchange operator) provides the equivalent regulatory basis. (Source: legislation.gov.uk/uksi/2013/609.)

T4 — fee rate: the specific fee rate (range: 15–25% per FinSim v4 sensitivity table) has not been tested with claimants. The central case (25% of £3,000 = £750/claim) is a financial model assumption, not a validated market preference. Pilot test design: present the fee rate to ≥20 pilot claimants during CP2 disclosure; measure proceed rate (target: ≥80% proceed, per PB-05). A proceed rate materially below 80% at the disclosed fee rate signals that the rate is at or above the market tolerance ceiling for this claimant population. FAIL threshold: <60% proceed rate at disclosed fee rate triggers fee rate review and FinSim v4 revalidation before scaling.
Acceptance criteria All four conditions must be met to declare PP-C verified:

AC1 — Transfer latency: ≥95% of net recovery transfers completed within 2 working days of settlement event, measured across ≥20 pilot settlements. FAIL consequence: payment rail engineering review — not an architectural return.

AC2 — Fee deduction accuracy: zero erroneous fee deductions across all pilot settlements (wrong fee rate applied, wrong gross settlement amount, or incorrect net calculation). Verified by automated reconciliation against the CP2-disclosed fee rate and PP-D settlement confirmation. FAIL consequence: platform engineering defect — payment logic to be corrected before any further settlements are processed.

AC3 — Claimant confirmation receipt: ≥95% of claimants receive transfer confirmation notification within 30 minutes of transfer completing, measured over pilot settlements. FAIL consequence: notification system engineering fix.

AC4 — Post-disclosure proceed rate (fee rate validation, T4): ≥80% of claimants who receive the CP2 fee rate disclosure proceed to pool admission (PP-C). Measured over ≥20 pilot claimants exposed to the full fee disclosure. This is not a PP-C internal metric — it is measured upstream at CP2 — but it is the acceptance criterion that validates the T4 fee rate assumption on which PP-C's revenue model depends. FAIL threshold: <60% proceed rate triggers fee rate review; <70% triggers a FinSim v4 sensitivity re-run before any scaling commitment.

Flow specification — PP-D: Payment flow — defendant fee collection

Derived from: design memo v2 (18 May 2026) — C6 defendant liability buyout architecture; FIT Verifier R6 (20 May 2026) — platform fee rate derivation, DBA regulatory analogue; FinSim v4 (21 May 2026) — revenue model, fee rate sensitivity table; Capella model — ICD-07 (D3 resolution event source → PP-D). Assumption tier T3 — payment-on-outcome mechanism grounded in UK DBA practice and third-party funder cost recovery; fee rate (% of settlement) is T4 pending pilot data.

Flow ID PP-D
Flow name Payment flow — defendant fee collection
Flow type Payment Product — the mechanism by which Calmly collects its defendant-side platform fee from the buyout operator at the D3 resolution event. PP-D is the primary revenue event in the Calmly architecture: no revenue is recognised before D3, and PP-D fires before PP-C (which distributes net recovery to the claimant from the same settlement proceeds). PP-D and PP-C are parallel payment flows operating on the same settlement transaction — PP-D extracts the defendant-side fee; PP-C extracts the claimant-side fee and transfers net recovery.
BALM sources C6 — No-win-no-fee architecture (defendant side): the defendant pays only at D3 resolution — the moment the buyout operator's payment clears into the exchange platform client account and the defendant's legal obligation is extinguished. No payment is due from the defendant or the buyout operator before this event. The payment-on-outcome structure is architecturally necessary: a defendant who faces a fee before resolution has a working capital problem before they have a settlement, which contradicts the Arrow (1963) risk-transfer logic that makes the buyout mechanism efficient. The fee is collected from the proceeds of the buyout payment, not in addition to it. Theory: Rubinfeld & Scotchmer (1993) — contingent fee structures eliminate liquidity barriers by aligning payment with value delivery; the same logic applies to the defendant's side as to the claimant's (Source: design memo v2, 18 May 2026; FIT Verifier R6, 20 May 2026).

Financial model driver (FinSim v4): PP-D is Calmly's primary revenue realisation event. At the central scenario (£3,000 average settlement, defendant fee rate × settlement amount), PP-D generates the gross platform fee income on which FMOS of 65.6% is calculated (FinSim v4, 21 May 2026 — sensitivity table, £3,000 / 25% cell). PP-D failure — non-collection at D3 — is the most asymmetric FMOS risk in the architecture: a single uncollected fee does not return to the cost floor calculation, meaning the practical collection rate must approach 100% for FMOS to hold as modelled. The target collection rate is ≥99% across all D3 events.
Carrier System (RC): Exchange Platform (OP1). PP-D is a fully automated payment flow triggered by the D3 resolution event (ICD-07). The exchange platform receives the buyout operator's payment, validates the D3 legal closure reference, applies the defendant fee extraction, and records the transaction — without human intervention. The Head of Finance (ARM: 0.5 FTE Stage 1, 1.0 FTE from Stage 2) provides compliance oversight of the client account and fee disbursement reconciliation; the payment event itself is system-automated. The buyout operator (FCA-regulated litigation finance firm — Burford Capital, Harbour, or equivalent) is the paying counterparty, not the defendant directly. From D3 onward, the defendant's obligation is to the buyout operator alone; Calmly has no further claim against the defendant.
Function On receipt of ICD-07 (D3 resolution event — defendant ID, settlement amount, legal closure reference, buyout operator payment confirmation), PP-D executes a two-step fee extraction:

1. Validate: confirm the D3 legal closure reference is valid (issued by the buyout operator, not a conditional agreement), the settlement amount matches the signed buyout agreement, and the payment has cleared into the exchange client account. If validation fails, PP-D does not fire — trigger exception path.
2. Extract: apply the Calmly defendant-side platform fee rate (a defined percentage of gross settlement amount) to the cleared funds. Record the transaction: claim ID, gross settlement amount, defendant fee rate applied, defendant fee amount, residual proceeds for PP-C, buyout operator ID, D3 legal closure reference, timestamp. Pass residual proceeds to PP-C for claimant fee extraction and net recovery transfer.

PP-D does not set the fee rate — that is an architecture-level parameter defined by the financial model (FinSim v4) and disclosed to both parties in the buyout agreement. PP-D enforces the rate; it does not negotiate it.
Behaviour Standard path — D3 resolution event received (buyout payment clears):
Trigger: ICD-07 fires from CP3 to PP-D — defendant ID, settlement amount, D3 legal closure reference, buyout operator payment confirmation.
Steps: (1) Exchange platform validates ICD-07 fields: legal closure reference authentic, settlement amount matches signed buyout agreement, payment cleared in client account. (2) Platform applies defendant fee rate to gross settlement amount: defendant fee = gross settlement × defendant fee rate. (3) Transaction record created: claim ID, gross settlement, defendant fee rate, defendant fee amount, residual proceeds, buyout operator ID, D3 closure reference, timestamp. (4) Residual proceeds flagged for PP-C processing. PP-C fires immediately after PP-D validation completes — both flows complete before any outbound transfer to claimant. (5) Reconciliation record sent to OP2 (C10 data infrastructure) and Head of Finance compliance log.

Exception path 1 — ICD-07 validation failure (payment not cleared or reference invalid):
Trigger: ICD-07 received but payment not yet cleared, or D3 legal closure reference does not match signed buyout agreement records.
Steps: (1) PP-D status set to PENDING_VALIDATION. (2) HQ Head of Finance alerted within 1 working hour. (3) Buyout operator contacted to resolve within 1 working day. (4) PP-D re-triggered when validation conditions are met. Claimant not notified until PP-C fires — PP-C must not fire before PP-D validation completes.

Exception path 2 — non-collection (buyout operator default):
Trigger: buyout operator payment does not clear within 5 working days of D3 legal closure reference issuance.
Steps: (1) CEO and Head of Finance alerted immediately. (2) Buyout operator formal demand issued via exchange platform legal channel. (3) If unresolved within 10 working days, matter referred to Calmly's legal advisor. FMOS asymmetry: this path is the highest-risk exception in the architecture — see Assumption Tier below.
Interfaces Receives via ICD-07 (inbound from CP3): D3 resolution event — defendant ID, settlement amount, D3 legal closure reference, buyout operator payment confirmation. Source: Capella model, ICD-07 — D3 resolution event from CP3 to PP-D. Trigger: D3 event (buyout operator signs repayment agreement and payment clears). Latency: PP-D fires within 15 minutes of ICD-07 received and validated.

Sends to PP-C (parallel payment flow): residual proceeds available for claimant fee extraction and net recovery transfer (gross settlement − defendant fee amount), claim ID, timestamp, D3 closure reference. No named ICD in the canonical Capella model for the PP-D → PP-C handoff — both flows operate on the same settlement proceeds held in the exchange client account. PP-C may not fire before PP-D validation completes.

Sends to OP2 (C10 data infrastructure): resolved claim record — claim ID, gross settlement, defendant fee amount, defendant fee rate applied, buyout operator ID, D3 timestamp, dispute category. Used for FMOS tracking and defendant fee rate validation against FinSim v4 assumptions.

Sends to Head of Finance compliance log: transaction record for FCA client account reconciliation (CASS rules).

No commingling with Calmly operating funds. The Head of Finance (ARM role) is the designated CASS oversight owner. Breach of this constraint is a regulatory event, not an operational one.

Sequencing with PP-C: PP-D must complete validation and fee extraction before PP-C fires. PP-C may not transfer net recovery to the claimant until the defendant fee has been extracted and recorded. The exchange platform enforces this sequencing constraint — PP-C is a dependent downstream event of PP-D, not an independent parallel trigger.

PP-D enforces the disclosed rate; it cannot vary per claim. Any change to the fee rate requires re-disclosure in the buyout agreement template and re-validation of FinSim v4 FMOS calculations.

UK litigation cost recovery (T3 regulatory): the mechanism by which a third-party funder extracts a fee at settlement is established in UK commercial litigation under the third-party funder cost recovery framework (LASPO 2012, s.58B; Arkin v Borchard Lines [2005] EWCA Civ 655 — proportionate funder cost recovery). Calmly's defendant fee collection at D3 is structurally analogous: the exchange platform extracts a fee from settlement proceeds at the resolution event. The specific regulatory basis for exchange-platform fee collection in the small claims context requires FCA authorisation review before launch — this is a constraint on timing, not a constraint on the mechanism itself.
Assumption tier T3 for the payment-on-outcome mechanism; T4 for the defendant fee rate and the ≥99% collection target.

T3 sourcing — payment-on-outcome mechanism: the collection of a fee from settlement proceeds at legal closure is the standard practice for third-party litigation funders in E&W (LASPO 2012; Arkin v Borchard Lines [2005]). The mechanism is legally operative and commercially tested in the UK litigation finance market. Calmly's PP-D is structurally analogous: fee extracted from proceeds at D3, before distribution to the claimant pool. The FCA authorisation framework (exchange operator) provides the regulatory basis. The specific terms for an exchange-platform operator (not a direct funder) require legal review — but the mechanism has established precedent.

The rate must be commercially acceptable to the buyout operator (who will factor it into their buyout pricing) and defensible to the defendant (who will see the total cost structure at D1). FAIL signal: buyout operator requires a reduction in the defendant platform fee to make the buyout offer price commercially viable — triggers FinSim v4 fee rate rebalancing between claimant and defendant lines.

T4 — ≥99% collection rate: the assumption that PP-D non-collection is rare relies on the buyout operator's payment obligation being legally enforceable and commercially certain at D3. The buyout operator pays the pool on signing the repayment agreement — not contingent on defendant repayment performance. This is an architecture-level assumption about the buyout operator's financial standing and contractual reliability. Pilot test design: review buyout operator credit rating and balance sheet before onboarding; include payment guarantee clause in the buyout operator framework agreement. FAIL threshold: if any pilot D3 event produces non-collection, the framework agreement is defective and must be revised before scaling.

AC2 — Validation latency: PP-D validation (ICD-07 received → fee extracted → residual proceeds flagged for PP-C) completes within 15 minutes of ICD-07 receipt, measured across pilot resolutions. FAIL consequence: exchange platform engineering review — does not require architectural return.

AC3 — Fee extraction accuracy: zero erroneous fee extractions (wrong fee rate applied, wrong gross settlement amount, or incorrect residual proceeds calculation) across pilot resolutions. Verified by automated reconciliation against the buyout agreement fee rate and ICD-07 settlement amount. FAIL consequence: payment logic defect — corrected before any further D3 events are processed.

Flow specification — CP2: EV disclosure and claim activation

Derived from: Capella model — ICD-04 (WP2 → CP2, EV assessment result), ICD-05 (CP2 → WP3, activated claim record); design memo v2 (18 May 2026) — C4 (pool survivability) and C5 (Coase assignment); FIT Verifier R5 (19 May 2026) — Coase assignment mechanism. Assumption tier T4 — post-EV proceed rate is the single most consequential untested commercial assumption in the CP2 architecture.

Flow ID CP2
Flow name EV disclosure and claim activation
Flow type Communications Product — the claimant-facing session that discloses the finalised EV figure (from WP2 via ICD-04), presents the structured path to legal closure, and obtains the claimant's commitment to proceed. On PASS, CP2 fires ICD-05 (activated claim record) to WP3 to initiate pool matching. CP2 is the last active decision point for the claimant in the architecture — after CP2, the Coase assignment (C5) moves all further litigation management to the pool operator. The claimant's active role ends here.
BALM sources C4 — Circumvent customer doubt about pool financial survivability (SR2): the claimant at CP2 has received an independent EV assessment confirming their claim is valid and quantified. The residual doubt is whether the pool will survive long enough — through early losses — for the expected value to materialise. CP2 addresses this directly: the disclosure session communicates the pool loss insurance mechanism (Part3) as a structural feature of pool membership. The insurance removes the individual incentive to exit early — the claimant's contribution is protected against early cohort losses regardless of pool short-term performance. Theory: Olson (1965) — collective action fails when individually rational defection undermines collective benefit; Arrow (1963) — insurance resolves the defection incentive by transferring idiosyncratic downside to a risk-pooling mechanism (Source: design memo v2, 18 May 2026).

C5 — Eliminate anticipated litigation complexity (Coase assignment): the claimant at CP2 carries a mental model of what litigation requires — court hearings, CPR compliance, legal documents, tactical decisions. This anticipated complexity is the primary barrier to proceeding after EV disclosure. CP2 resolves this by communicating — explicitly and verifiably — that the claimant's active role ends at this session. All post-commitment litigation management is assigned to the pool operator as professional litigant. Theory: Coase (1937), Williamson (1985) — activities assigned to the party with lowest total transaction cost; the pool operator's specialist assets (CPR expertise, court familiarity, economies of scale across a claims portfolio) make the assignment economically necessary, not optional. The Coase assignment is not a service tier — it is communicated as a structural default of pool membership (Source: FIT Verifier R5, 19 May 2026; design memo v2, 18 May 2026).
Carrier Role (PVC): EV Specialist. ARM role: EV assessment specialist at LMU level. The EV specialist who conducted quality review of the WP2 report carries CP2 — the same individual who validated the figure discloses it to the claimant, providing continuity of expertise and eliminating source ambiguity. Per-session time budget: ≤60 minutes (ARM-derived, including EV disclosure, structured path presentation, C4/C5 communication, and commitment confirmation). Session modality: video call or phone call — the EV specialist calls the claimant at the scheduled time. The session is not self-service: human presence is the carrier for both the EV figure's credibility and the Coase assignment's psychological weight. Replacing the specialist with a digital-only flow is a design choice that would require separate BALM verification — it is not the current architecture.
Function CP2 performs four sequential sub-functions in a single session:

1. Disclose: present the finalised EV figure (ICD-04 — £ low/mid/high range) to the claimant. Explain the methodology without prejudice to defendant. Confirm the figure is independent and not produced by Calmly. Confirm without-prejudice cover applies.
2. Present path: walk the claimant through the structured path to legal closure — pool matching (WP3), defendant engagement (CP3), D1–D3 resolution sequence. Communicate the timeline (ARM-derived: 45-day resolution target for settling defendants). Present the fee structure: claimant platform fee (% of gross settlement, deducted at PP-C), pool insurance as standard feature (C4). Make clear that all post-commitment litigation management is the pool operator's responsibility (C5 Coase assignment).
3. Obtain commitment: confirm the claimant wishes to proceed. Record commitment electronically (exchange platform confirmation screen, timestamped). This is the trigger for ICD-05. If the claimant declines, CP2 FAIL — claim record archived as abandoned; no WP3 routing.
4. Activate: on commitment confirmed, fire ICD-05 (activated claim record — claim ID, EV figure, claimant commitment, timestamp) to WP3 for pool matching.

Constraints Without-prejudice constraint (regulatory): the EV figure disclosed in CP2 is produced under without-prejudice cover — it cannot be used as evidence of admission in any subsequent proceedings. The EV specialist must communicate this explicitly in the session. The disclosure script (ARM-derived) includes the without-prejudice framing; the exchange platform confirmation screen records the claimant's acknowledgment of this constraint before commitment confirmation.

Fee disclosure timing: the full fee structure (claimant platform fee rate, pool insurance cost structure) must be disclosed to the claimant in CP2 before commitment confirmation is recorded. CP2 is the contractually binding disclosure moment — not CP1, not any subsequent communication. Any change to the fee rate requires a CP2 disclosure script update before the next session is conducted.

C4 constraint (no premature EV disclosure): the EV figure must not be shared with the claimant before CP2. CP1 intake explicitly withholds indicative valuations (C4 Constraint 4 from the WP1 spec and CP1 spec). CP2 is the first and only session at which the figure is disclosed. Premature disclosure risks consumer financial promotions rule violations (FCA CONC perimeter) and compromises C2's motivated-reasoning disruption mechanism by giving defendants grounds to challenge the disclosure protocol.

Session modality (human carrier): CP2 must be conducted by a qualified EV specialist in real time (video or phone). A digital-only session (automated EV report delivery without specialist presence) is not the current architecture — it would require separate BALM re-verification of C4 and C5 before adoption. This constraint bounds the ARM headcount model: CP2 sessions drive EV specialist time demand at ≤60 minutes per claim, which is the ARM-modelled rate.
Assumption tier T4 — post-EV proceed rate is the single most consequential untested commercial assumption in the CP2 architecture. All other T4 elements in the system can fail without immediately invalidating FMOS — PB-05 failure invalidates the revenue model directly.

PB-05 — post-EV proceed rate: the probability that a claimant who receives the CP2 EV disclosure proceeds to pool admission. Modelled as PERT-Beta(0.66, 0.80, 0.90) — minimum 66%, most likely 80%, maximum 90%. The 80% central case is the architecture's working assumption. The assumption is grounded in analogous contexts (Citizens Advice referral conversion rates; no-win-no-fee legal services take-up post-quote) but has not been tested in this specific setting. FAIL consequence: at a 66% proceed rate, 34% of claims complete WP2 (cost incurred) and generate zero revenue — raising the effective per-claim cost and compressing FMOS. A proceed rate materially below 70% requires FinSim v4 re-run before any scaling commitment; below 60% is an architectural signal that CP2's C4/C5 communication is failing to resolve the commitment barriers.

Pilot test design (required before scaling): conduct ≥20 live CP2 sessions with real claimants who have received a real WP2 EV assessment. Measure: (1) proceed rate (ICD-05 PASS rate); (2) reasons for abandonment (structured post-session survey for abandoned claimants); (3) claimant comprehension of the Coase assignment and fee structure (unprompted recall at 30-day check-in, target ≥80% accuracy). The pilot test cannot be simulated — it requires live WP2 assessments upstream, which means WP2 must be piloted before CP2 can be tested. Build sequence consequence: WP2 is a prerequisite for CP2 piloting, not a parallel track.

T4 secondary — session length (≤60 minutes): the ARM-modelled ≤60-minute session budget is untested in practice. A session that consistently runs to 90 minutes would increase EV specialist labour cost and ARM headcount per LMU by ~50%, raising the cost floor. Monitor in pilot; adjust ARM if exceeded systematically.
Acceptance criteria All four conditions must be met to declare CP2 verified:

AC1 — Post-EV proceed rate (PB-05): ≥80% of claimants who receive a CP2 session commit to proceed (ICD-05 fires), measured across ≥20 pilot sessions. FAIL threshold: <70% triggers FinSim v4 re-run; <60% is an architecture signal — CP2 communication design reviewed before further sessions. This is the primary test: it is the only acceptance criterion that, if failed, immediately invalidates the FMOS model.

AC2 — Coase assignment comprehension: ≥80% of claimants who proceed correctly understand (unprompted, at 30-day check-in) that they will not need to attend court or manage legal documents after CP2. FAIL consequence: CP2 session script redesigned — Coase assignment communication is the C5 mechanism; an AC2 fail means C5 is unverified in this flow.

AC3 — Session scheduling latency: ≥90% of CP2 sessions scheduled within 48 hours of ICD-04 receipt and conducted within 5 working days. FAIL consequence: ARM scheduling model review — does not trigger architectural return.

AC4 — Without-prejudice acknowledgment rate: 100% of claimants who proceed have their without-prejudice acknowledgment recorded on the exchange platform confirmation screen before ICD-05 fires. Zero exceptions permissible. FAIL consequence: platform confirmation screen engineering fix; sessions suspended until resolved.

Flow specification — Part3: Pool insurance marketplace integration

Derived from: Capella model — Part3 function description (BALM C5, T2, off-balance-sheet); design memo v2 (18 May 2026) — C4 pool loss insurance mechanism, Olson (1965), Arrow (1963); FIT Verifier R5 (19 May 2026) — Coase assignment and pool operator structural default. No direct claim-chain ICD — Part3 is a system-level flow connecting pool operators to FCA-regulated insurers; it is not triggered by a claim event. Assumption tier T2 — general insurance marketplace regulation (ICOBS) is established; specific insurer willingness to quote pool loss risk is T3 pending market engagement.

Flow ID Part3
Flow name Pool insurance marketplace integration
Flow type Partner Product — an exchange marketplace connecting pool operators to FCA-regulated insurance products for pool loss coverage. Calmly acts as marketplace operator, not insurer: it does not underwrite pool loss risk, it does not hold the insurance on its own P&L, and it does not guarantee pool operator claims against the insurer. The insurance premium is the pool operator's cost, not Calmly's. Calmly earns a marketplace curation fee from the insurer for each policy placed. The pool loss insurance is communicated to claimants as a structural feature of pool membership (C4) — but the contract is between the pool operator and the FCA-regulated insurer, intermediated by the exchange marketplace.
BALM sources C5 — Eliminate anticipated litigation complexity (pool insurance available off P&L): pool insurance available through the exchange marketplace means the pool operator can purchase coverage that protects member contributions against early losses — without that cost sitting on Calmly's balance sheet. This is the architectural mechanism that allows Calmly to communicate pool loss insurance as a standard feature of pool membership (C4) without bearing insurance liability directly. The Coase assignment (C5 primary mechanism) assigns litigation management to the pool operator; Part3 assigns pool loss risk to the insurer. Both assignments are off-P&L for Calmly: the pool operator manages litigation, the insurer absorbs pool loss volatility. Source: Capella model, Part3 BALM C5 annotation; design memo v2, 18 May 2026.

C4 — Circumvent customer doubt about pool financial survivability (enabling mechanism): without Part3, pool loss insurance cannot be offered as a standard feature of pool membership without Calmly bearing the underwriting risk directly. Calmly is not an insurer and cannot carry unlimited pool loss liability on its P&L without destroying FMOS. Part3 is the mechanism that makes C4 structurally viable: the insurer bears the risk; the pool operator purchases the product; Calmly communicates the protection to claimants as a standard structural feature. If Part3 fails to deliver an insurer willing to quote pool loss risk at an affordable premium, C4's pool survivability assurance cannot be provided as an architectural default — and the claimant confidence mechanism breaks. Part3 is therefore not optional: it is a structural dependency of C4. Theory: Arrow (1963) — efficient risk allocation to the party with lowest cost of bearing it; a diversified insurer bears pool loss risk at lower cost than either Calmly or individual claimants (Source: design memo v2, 18 May 2026).

C6 — Pool operator participation reduces cash flow exposure: a pool operator who can purchase insurance for pool losses faces a bounded, insurable downside rather than an open-ended WC exposure from adverse claim outcomes. This improves pool operator willingness to participate (C3 pool operator recruitment, Part2) and reduces the WC premium the operator would otherwise demand in their claim purchase pricing. Part3 therefore has a secondary positive effect on the C3 architecture — it is not just a C4/C5 mechanism.
Carrier Primary carrier — Role (SC): HQ Marketplace Manager. ARM role: Head of Marketplace Partnerships (HQ level). Responsible for recruiting FCA-regulated insurers to the exchange marketplace, negotiating framework insurance terms, and managing the ongoing insurer relationship. Per-policy time budget: approximately 2–4 hours for policy issuance and annual renewal (ARM-derived from analogous insurance distribution models). Volume: one policy per active pool operator per year (not per claim) — Part3 is not a per-claim flow, it is a per-operator annual cycle.

Secondary carrier — Third-party: FCA-regulated Insurer. The underwriter of pool loss risk. Analogues: Lloyd's of London syndicates (specialist litigation risk); ARAG or DAS (legal expenses insurers with existing UK litigation portfolio exposure); Allianz Legal Protection. The insurer operates on its own P&L — Calmly is the marketplace intermediary, not the risk-bearing party. Regulatory basis: the insurer must be FCA-authorised for general insurance (ICOBS applies to all non-investment insurance products distributed in the UK). Calmly's marketplace role triggers FCA insurance distribution obligations — specifically ICOBS rules on fair and accurate disclosure and suitability of insurance recommendations. Calmly must be FCA-authorised or appoint an FCA-authorised distributor before Part3 goes live.
Function Part3 operates on two timescales — setup (one-off per insurer relationship) and ongoing (per-operator annual cycle):

Setup (insurer recruitment): HQ Marketplace Manager recruits ≥1 FCA-regulated insurer to the exchange marketplace. Negotiates a framework insurance product covering pool loss risk (claim cohort early-loss protection). Framework terms include: premium basis (per-pool-operator per year, calculated as a % of pool operator's expected claim portfolio value), coverage trigger (loss exceeding a defined % of expected portfolio value), policy limit (capped at a defined multiple of annual premium), exclusions. Framework agreed and executed as a signed policy schedule before any pool operator is onboarded under Part3.

Ongoing (per-operator annual cycle): each pool operator who joins the exchange (via Part2 onboarding) is offered a pool loss insurance policy via the marketplace. The HQ Marketplace Manager presents the policy terms; the pool operator decides whether to purchase. A pool operator who purchases receives a certificate of coverage; Calmly records the coverage in the QA Tracking System (feeding Part2's operator profile). The policy renews annually; HQ Marketplace Manager manages renewal and any mid-term adjustments. Calmly communicates pool operator coverage status to claimants as the basis for the C4 pool survivability assurance.
Behaviour Setup path — insurer recruitment (one-off):
Steps: (1) HQ Marketplace Manager produces pool loss insurance proposition pack: pool portfolio composition (claim category split, EV distribution, historical acceptance rate once available), projected annual claim volume, target coverage structure. (2) Outreach to ≥3 FCA-regulated insurer candidates (Lloyd's syndicates, ARAG, Allianz Legal Protection, or equivalent). (3) Framework negotiation: premium basis, coverage trigger, policy limit, exclusions agreed. (4) Framework insurance policy signed and loaded to exchange platform. (5) Part3 marketplace is live — pool operators can be offered coverage from the date the framework policy is executed.

Ongoing path — per-operator policy issuance:
Trigger: new pool operator onboarded via Part2 (signed framework agreement, routing weights set in WP3).
Steps: (1) HQ Marketplace Manager presents pool loss insurance policy terms to new operator within 5 working days of Part2 onboarding completion. (2) Operator reviews terms and elects to purchase or decline. (3) On purchase: policy certificate issued (insurer → operator → exchange platform record). QA Tracking System updated: operator coverage status = INSURED. (4) On decline: QA Tracking System updated: operator coverage status = UNINSURED. HQ Marketplace Manager notes the decline; this does not prevent WP3 routing but is flagged in the C4 communication review (Calmly may not communicate pool insurance as a universal structural feature if ≥1 active operator is uninsured — see Constraints below). (5) Annual renewal: HQ Marketplace Manager initiates 60 days before policy expiry; renewal terms reviewed; operator renewal confirmed or lapsed.

Exception path — insurer unwilling to quote at affordable premium:
Trigger: all three insurer candidates decline to quote or quote at a premium that makes pool operator participation uneconomic (premium > 5% of expected claim portfolio value per year — HQ Marketplace Manager threshold, not yet validated).
Steps: (1) HQ Marketplace Manager reports to CEO within 5 working days. (2) Alternative coverage structures reviewed: excess-of-loss rather than first-loss; captive structure with reinsurance backstop. (3) If no viable structure found within 60 days, C4 communication reviewed — Calmly cannot communicate pool loss insurance as a standard structural feature if no insurer can be recruited. Architectural implication: C4's pool survivability assurance must be redelivered through an alternative mechanism. This is a design-return signal, not an operational fix.
Interfaces No direct claim-chain ICD: Part3 does not receive or send claim-level ICDs. It operates at the pool operator level, not the claim level. The Capella canonical model records no ICD for Part3 — consistent with its system-level, non-claim-triggered function.

Sends to Part2 QA Tracking System: pool operator coverage status (INSURED / UNINSURED), policy certificate reference, coverage period, insurer ID. Trigger: policy issuance or renewal. Used by Part2 to maintain complete operator profiles and by WP3 to flag any routing decisions to uninsured operators for HQ awareness.

If all active operators are insured, CP2 communicates pool loss insurance as a standard structural feature of pool membership. If ≥1 operator is uninsured, HQ Marketplace Manager updates the CP2 disclosure script to reflect actual coverage status. This is not an automated interface — it is a governance dependency between Part3 and CP2.

Reads from Part2 operator profiles: pool operator portfolio composition (claim category, volume, EV distribution) to build the insurance proposition pack for insurer recruitment. Updated on each Part2 quarterly monitoring cycle.
Constraints FCA insurance distribution (ICOBS): Calmly's role as marketplace intermediary between pool operators and insurers triggers FCA insurance distribution obligations under ICOBS (Insurance: Conduct of Business Sourcebook). Specifically: ICOBS 2 (fair, clear, and not misleading communications); ICOBS 5 (demands and needs — ensuring the insurance is suitable for the pool operator's needs); ICOBS 6 (disclosure of commission, including any curation fee Calmly earns from the insurer). Calmly must be FCA-authorised as an insurance intermediary or appoint an authorised AR before Part3 goes live. This constraint is a timing gate, not a design constraint — the mechanism is legally viable; the authorisation timeline is the constraint.

If ≥1 active operator is uninsured, the CP2 C4 communication must be qualified accordingly. This constraint binds Part3 to CP2's disclosure script — a governance dependency that must be managed on every policy issuance, lapse, and renewal cycle.

Insurer counterparty concentration: if only ≥1 insurer is on the marketplace, Calmly faces single-source counterparty risk — the insurer withdraws, revises terms, or becomes insolvent, and C4 fails without warning. Target: ≥2 insurers on the marketplace before Calmly exits pilot. One insurer is minimum viable for pilot phase; two is the structural target for scaling. This is the Part3 equivalent of Part2's ≥3 pool operator constraint.

The HQ Marketplace Manager monitors acceptance rate on Part3 policy offers. Target: ≥80% of pool operators elect to purchase coverage. A below-target acceptance rate indicates the premium is above the pool operator's affordability threshold and requires renegotiation with the insurer.
Assumption tier T2 for the FCA general insurance marketplace mechanism; T3 for insurer willingness to quote pool loss risk at an affordable premium.

Lloyd's of London syndicates, ARAG, DAS, and Allianz Legal Protection all operate through exactly this intermediary model. However, no insurer has been approached and no quote has been obtained. Flow ID OP1
Flow name Exchange platform (core technology)

Function OP1 performs four persistent infrastructure functions — not triggered by individual claim events, but running continuously as the platform operates:

3. Behaviour OP1 does not have a claim-level state machine — it is always running. Its behaviour is characterised by availability and throughput, not by per-claim transitions.

Normal operation: platform available, all routing, matching, and payment rail functions operating within latency budgets. Flow ID OP2
Flow name EV methodology and assessment engine
Flow type Operational — the methodology stewardship and data accumulation layer that owns Calmly's EV assessment methodology, governs its accuracy over time, and builds the outcomes dataset that constitutes Calmly's primary competitive moat (C10). OP2 does not execute individual claims — it maintains and improves the methodology that WP2 executes. OP2 is the difference between a static methodology (which competitors can copy once published) and a compound accuracy advantage (which competitors cannot copy because it requires years of calibration data). The flow is continuous and system-level: it receives ICD-11 data points from every completed WP2 assessment, accumulates them into a calibration dataset, and produces methodology updates at defined intervals. OP2 is the engine behind C10; C10's moat exists only as long as OP2 is actively maintained.
BALM sources C1 — Methodology keeps assessment cost credible: the EV methodology determines what assessment orgs are required to produce and how they are evaluated. A methodology that is rigorous enough to be defensible in a without-prejudice context (so defendants take it seriously) but structured enough to be executable by university law students (so it remains within the £16.28/claim cost ceiling) is not a one-time design artefact — it requires ongoing maintenance. As claim categories expand, as case law evolves, and as assessment org accuracy data accumulates, the methodology must be updated to remain calibrated. If the methodology drifts — either becoming too demanding (cost ceiling breach) or too permissive (accuracy loss) — C1's cost-floor economics break. OP2's methodology governance function is the mechanism that keeps C1 stable over time. Source: Capella OP2 BALM annotation; FIT Verifier R10 section 5 (methodology IP as secondary moat).

C10 — Proprietary outcomes dataset accretes accuracy advantage with volume: every resolved claim produces a data point — EV prediction (£ mid figure, from ICD-11) matched against actual settlement outcome (recorded retroactively when the claim resolves). The gap between prediction and outcome is the accuracy signal. As the dataset grows, Calmly can (a) identify systematic biases in methodology (e.g., housing category predictions consistently 15% above settlement — addressable with a calibration factor); (b) improve confidence interval tightness (£ low/mid/high range narrows as the dataset provides more reference class observations); (c) identify defendant response patterns (early acceptance rate by claim category and value band — improves pool operator pricing and buyout operator underwriting). The dataset is the moat, not the methodology document. A competitor can read the published methodology; they cannot replicate 5 years of calibration data. Source: design memo v2 C10; FIT Verifier R10 — compound accuracy advantage, data flywheel, Metcalfe's law (as heuristic). Theoretical basis: Kahneman and Tversky (1979) reference class forecasting — outside-view accuracy improves systematically with reference class dataset size; Tetlock (2005) Superforecasting — calibrated forecasters outperform uncalibrated forecasters by a margin that grows with the breadth of outcome feedback. Named analogue: Rightmove Automated Valuation Model (AVM) — accuracy improvement over 20 years of UK property transaction data. Rightmove's AVM produces tighter confidence intervals than any competitor's model on UK residential property because it has 20+ years of indexed transaction outcomes. Calmly's EV dataset for consumer disputes is the same structural asset in a different domain.

Function OP2 performs three continuous functions:

1. Ingest and store (ICD-11 receipt): on each WP2 PASS event, receive ICD-11 (EV prediction — £ mid figure, claim category, ITP source, assessment org ID, methodology version). Store in the EV outcomes dataset with a null outcome field (outcome not yet known — claim is in progress). When the claim resolves (CP3 D3 event), update the record with the actual settlement amount. The prediction-vs-outcome pair is then available for accuracy analysis.
2. Compute and report (accuracy analysis): at defined intervals (monthly at pilot scale), compute: (a) mean absolute percentage error (MAPE) between £ mid prediction and actual settlement, by claim category; (b) confidence interval accuracy — proportion of actual settlements falling within the £ low/£ high range; (c) systematic bias indicators — is one claim category or one assessment org consistently over- or under-predicting? Produce a written accuracy report distributed to the EV methodology analyst and HQ Methodology Team.
3. Update and version (methodology governance): when accuracy analysis identifies a systematic bias or a methodology gap (new claim category, case law change), the HQ Methodology Team produces a methodology update. Version history is maintained as Calmly's IP — it cannot be transferred or accessed by assessment orgs outside the scope of the contract. A methodology update that changes EV figures materially (i.e., produces assessments more than ±10% different from the previous version on the same claim) requires re-verification against the WP2 AC1 accuracy test before deployment.
Behaviour Standard operation (ICD-11 ingestion — continuous):
Trigger: WP2 PASS event fires ICD-11. Latency: ICD-11 ingested and stored within 15 minutes of receipt. Outcome field populated with NULL (pending). Record available for accuracy analysis from time of ingestion. No human action required at ingestion — automated pipeline.

Retroactive outcome recording (CP3 D3 event):
Trigger: CP3 records a D3 resolution event (defendant signs buyout agreement; settlement amount confirmed). Platform routes the settlement amount to OP2 alongside the claim ID. OP2 matches the settlement amount to the stored ICD-11 prediction record by claim ID. Prediction-outcome pair marked COMPLETE. The latency from settlement event to OP2 record update: ≤1 working day (automated, no human review required).

Monthly accuracy report (HQ Methodology Team):
Trigger: first working day of each calendar month. EV Data Infrastructure computes accuracy metrics across all COMPLETE prediction-outcome pairs in the dataset. MAPE, CI accuracy rate, and systematic bias indicators written to the methodology dashboard. HQ EV methodology analyst reviews within 5 working days. If no systematic bias is detected above the threshold (MAPE >15% for a claim category with ≥10 data points), no methodology update is triggered — the analyst records "no action required" and the report is filed.

Methodology update cycle (triggered by accuracy report):
Trigger: accuracy report identifies a systematic bias above threshold (MAPE >15% on ≥10 data points in a claim category, or CI accuracy rate <80% system-wide). HQ Methodology Team produces a draft methodology update within 20 working days. Draft reviewed by HQ Legal for without-prejudice compatibility and IP custody compliance. Version released; assessment orgs notified; 30-day transition window applies. Methodology update logged in version history with accuracy trigger data and rationale.

Accuracy reports in Stage 1 are directional — they report the raw prediction-outcome pairs without statistical significance claims. Methodology updates in Stage 1 are triggered by qualitative review (e.g., assessors consistently missing a class of claim type) not by statistical thresholds. Dataset completeness (retroactive outcome recording): the accuracy analysis is only as good as the completeness of the prediction-outcome dataset. If CP3 resolution events are not reliably fed back to OP2 (e.g., claims that settle informally outside the exchange platform, or claims that reach court judgment without being recorded), the dataset has survivorship bias — it records only the claims that went through the full platform pathway. At scale, automated reconciliation is required: the platform must record all D3 events (buyout signing) and all court judgment events as settlement triggers that feed OP2. At Stage 1, the threshold is operationally moot (insufficient data). At Stage 2, the threshold must be reviewed by HQ Methodology Team against actual data distribution before being deployed as an automated trigger. A threshold that is too sensitive (MAPE >5%) would trigger methodology updates after every small data fluctuation — destabilising assessment orgs. A threshold that is too permissive (MAPE >30%) would allow systematic biases to compound before correction. The threshold is a governance parameter that requires human judgment at each review cycle, not a fixed algorithmic rule.

Dispatch timestamp logged; defendant receipt tracking enabled (email open, postal delivery confirmation).
2. Escalation state management (D1 → D2 → D3): for each claim in the defendant engagement workflow, OP3 tracks the D1/D2/D3 state and fires escalation events at defined trigger points. D1 to D2: if no defendant response within the D1 window (14 to 21 days — per C2 design), fire D2 escalation notification (reminder with updated escalation schedule — the settlement ask has increased by one stage of legal cost accumulation). D2 to D3: if no response within the D2 window (further 14 days), route to defendant engagement specialist for human escalation review. D3 PASS: defendant signs buyout agreement — fire ICD-07 (D3 resolution event) to PP-D for settlement processing. D3 FAIL: defendant does not sign — route to D4 (court proceedings pathway, managed by pool operator as professional litigant).
3. Human escalation routing: for non-response, dispute, and negotiation cases (D3 human mediation), OP3 routes the case file to the defendant engagement specialist with a summary: claim ID, EV figure, D1/D2 timeline, defendant contact history, dispute type. The specialist takes over from the automated workflow; OP3 records all specialist actions and timestamps for FCA audit trail. On specialist resolution (settlement agreed), specialist records D3 PASS in OP3; ICD-07 fires automatically. On specialist failure (no settlement), specialist records D3 FAIL; D4 pathway is activated.
4. Audit trail maintenance: OP3 maintains a complete, tamper-evident log of all defendant communications — dispatch timestamps, open/delivery confirmations, response events, escalation triggers, specialist actions. This log is the evidentiary basis for CPR compliance (demonstrating that the defendant received adequate notice before court proceedings) and FCA audit (demonstrating that defendant communications complied with Consumer Protection from Unfair Trading Regulations 2008 and FCA conduct requirements). Log retention: minimum 7 years from claim resolution date.

Constraints Without-prejudice legal framework: the D1 notification package — specifically the EV settlement report and the conditional buyout offer — is communicated under without-prejudice cover. This means: (a) the EV figure and the settlement offer cannot be used as evidence of admission in any court proceedings if negotiations fail; (b) the D1 letter must include explicit without-prejudice marking and the defendant must be informed of its meaning; (c) if OP3 dispatches communications that do not include the without-prejudice marking, the legal protection is lost — any subsequent court proceedings would be prejudiced by the prior notification. Constraint: the D1 notification template must be reviewed and approved by legal counsel before first live dispatch. Every subsequent template version that changes the EV disclosure or settlement offer wording requires legal review before deployment. Source: Walker v Wilsher [1889] (without-prejudice rule); Cutts v Head [1984] (without-prejudice privilege); CPR Practice Direction 36.

Consumer Protection from Unfair Trading Regulations 2008 (CPRs): defendant notification is a commercial communication to a consumer. The CPRs prohibit: (a) misleading actions — stating the EV figure as a guaranteed outcome rather than a probabilistic assessment; (b) misleading omissions — failing to disclose the basis of the EV figure, the identity of the assessment org, or the fee structure; (c) aggressive commercial practices — pressuring the defendant by creating a false sense of urgency, making threats that cannot be carried out, or using language that could constitute harassment. OP3's notification templates must be audited against the CPRs before first live dispatch. The escalation timeline (D1: 14 to 21 days; D2: 14 days) must be designed to give the defendant a reasonable and genuine opportunity to respond — not a manufactured urgency. Legal counsel review required before any template is deployed. Source: Consumer Protection from Unfair Trading Regulations 2008 SI 2008/1277; Guidance on the Consumer Protection from Unfair Trading Regulations — CMA (2016).

CPR compliance (service of process): the D1 notification package is not a formal court document, but it initiates the pre-action protocol that precedes any D4 court filing. CPR Pre-Action Protocols require that the defendant receive reasonable notice of the claim and an opportunity to respond before proceedings are issued. OP3's tracked postal dispatch provides the evidentiary record of service. If the defendant is unreachable, CPR 6.15 (service by alternative means) may allow service via ITP or other channel — but this requires court permission and cannot be assumed. The DEFENDANT_UNREACHABLE exception path must include legal review before D4 is activated, not after.

OP3's notification templates are designed to comply with CONC 7 as a minimum standard (even though the defendant in a consumer dispute claim is not technically a "debtor" under CONC 7 — they are a defendant in a civil claim). The CONC 7 framework provides the consumer protection reference point for template design. (Source: FCA Handbook CONC 7 — Arrears, default, recovery.)

T3 caveat — defendant response rate: the proportion of defendants who respond favourably at D1 (accept buyout offer within the D1 window without escalation) is the primary untested assumption in OP3. Design memo v2 models the defendant threshold mechanism based on Arrow (1963) risk transfer theory — defendants rationally prefer legal certainty (D3 buyout) to open-ended litigation risk when the buyout price is set at or below EV. The CIVEA analogue (60–70% resolution at pre-enforcement stage) is directional support, but the claim type, financial context, and communication channel are different. The pilot must measure D1 resolution rate directly. FAIL threshold: if D1 resolution rate is below 40% (i.e., more than 60% of defendants require D2 or D3 escalation), the OP3 cost model and ARM specialist headcount require revision before scaling — the human escalation cost is material if the D1 automated path is not the majority case.

Pilot test design (required before scaling): conduct ≥20 live D1 notifications (real defendants, real claims matched via CP3) and track: (1) D1 resolution rate (defendant accepts within window, no escalation required); (2) D2 escalation rate; (3) D3 human escalation rate; (4) D3 resolution rate (of those reaching human escalation, proportion who settle vs proceed to D4); (5) average elapsed time from D1 dispatch to D3 PASS. These five metrics are the inputs to the ARM specialist headcount model and the FinSim v4 resolution timeline assumption (45-day target). The pilot cannot be simulated — it requires live WP3 pool matches upstream, which requires live CP2 activations, which requires live WP2 assessments. D1 notification pilot is the longest-dependency validation in the architecture: it requires every upstream flow to be operational before it can be tested.

BALM source C1 — Cost ceiling enforcement: OP4 is the operational mechanism that prevents any single org from becoming entrenched. A monopoly assessment org would lose the competitive accuracy pressure that keeps per-claim cost at ≤£16.28 (ARM ceiling). If one org handles all claims without competitive alternatives, cost negotiation leverage is zero. OP4's concentration ceiling (≤30% per org) is therefore a cost-floor protection mechanism, not merely a supplier diversity preference.

C9 — Structural replaceability: the concentration ceiling (≤30% per org in any rolling 90-day window) and minimum pool size (≥3 active orgs) are the specific C9 mechanisms. C9's logic: supplier replaceability must be structurally enforced, not merely intended. An org that handles 60% of volume cannot be replaced in <10 working days without service disruption — the pool must be maintained at sufficient diversity that any single org's exit is absorbed without a claim processing pause.
Function Four sub-functions:
1. Score: ingest ICD-10 data from each WP2 claim completion; update rolling accuracy score (% of reports passing quality review first time), dispute rate, and turnaround SLA compliance per org.
2. Allocate: update routing weights for WP2 claim dispatch based on current org scores; highest-quality available org (by composite score) receives next available claim batch, subject to the ≤30% concentration ceiling.
3. Enforce: if any org's rolling 90-day volume share approaches 28% (trigger threshold), alert HQ QA Manager; if it breaches 30%, freeze new allocations to that org until the window resets or the pool is expanded.
4. Replace: if an org's accuracy score falls below the minimum standard, or if the pool falls below ≥3 active orgs, trigger the replacement process — HQ QA Manager activates a pipeline org within ≤10 working days.
Behaviour Standard path — ICD-10 received: QA tracking system ingests report data; accuracy score and concentration % recalculated; routing weights updated automatically; no human action required if all thresholds within bounds.

Concentration alert: any org's 90-day share approaches 28% → automated alert to HQ QA Manager → manager reviews pipeline org readiness and adjusts allocation guidance before the 30% ceiling is reached. No claim processing disruption.

Concentration breach: any org's 90-day share exceeds 30% → new allocations to that org frozen → claims rerouted to next-ranked org in pool → HQ QA Manager notified immediately → pool rebalancing plan produced within 2 working days.

Accuracy breach: org's rolling quality review pass rate falls below 80% (first-pass) across the most recent 30 claims → HQ QA Manager initiates improvement dialogue → if not resolved within 15 working days, org is suspended from pool and replacement activation begins → replacement org operational within ≤10 working days.

Pool size breach: active qualified org count falls to 2 → HQ QA Manager activates pipeline org immediately (target: ≤5 working days to provisional activation, ≤10 working days to full QA clearance).

Creditors cite StepChange data in their own regulatory filings (FCA CONC reporting) and annual reports. The mechanism is structurally identical to C8: the outcomes data becomes embedded in the creditor's accountability framework, creating a switching cost independent of contract terms.

The T3 rating reflects that this analogue operates in the debt advice context, not the dispute resolution context. The ITP types in the Calmly architecture (Citizens Advice, law centres, housing associations) are analogous to but not identical to StepChange's creditor partners. The switching cost magnitude — how much the ITP's stakeholders actually care about the outcomes data — is not validated.

Pilot test design: AC1 (above) is the specific pilot test. It must be measured at 12 months, not inferred from report delivery alone. If AC1 is not met at 12 months, the T3 assumption is falsified and the C8 mechanism requires redesign.

BALM source C9 — Structural replaceability (geographic diversification of regulatory risk): OP6 creates local redundancy at territory level. The structural argument: a platform operating in a single territory faces existential regulatory risk from any adverse regulatory change in that territory. A platform operating across multiple territories with local compliance infrastructure faces territory-specific regulatory risk that is contained — one territory's adverse event does not propagate to others. OP6 is the local compliance infrastructure that makes this geographic diversification real rather than nominal.

Description OP7 is the ineligibility notification dispatch flow. It fires when WP1 marks any eligibility criterion FAIL and is responsible for routing a no-proceed notification to the claimant. It crosses a system boundary — the claimant is an external actor — and therefore requires a named flow spec rather than treatment as an internal WP1 event.
Trigger WP1 assessor marks any eligibility criterion FAIL. ICD-09 fires: WP1 sends claimant contact details, fail reason code (E1–E6), plain-language explanation, and referral signpost to OP7. Latency: notification sent within 1 working day of FAIL determination.

ICDs ICD-09 (inbound): WP1 → OP7. Data: claimant contact details, fail reason code (E1–E6), plain-language explanation, referral signpost flag. Trigger: FAIL determination in WP1. Latency: ≤1 working day.

A claimant who understands why they were ineligible is less likely to form a negative impression of Calmly and more likely to return if their circumstances change. This is a reputational contribution to C4, not a primary mechanism. OP7 does not appear in the RTM as a C4 carrier — it is noted here as a secondary effect.

AC2 — Referral signpost present: 100% of notifications include at least one referral option. Measured by template audit and automated content check.

AC3 — Plain-language compliance: zero legal terminology in notifications. Measured by template review at build time; no runtime audit required if template is fixed-content.

Sequence the build

Order flows by risk first, dependency second. Design, prototype (if needed), and test the highest-risk flow clusters first — so architectural failures surface before downstream capital is spent.

The risk-first principle: a T4 FAIL is not an execution problem — it is an architectural signal. Return to the BALM requirement responsible. A T4 FAIL discovered after 6 months of build is expensive. Discovered first, it is information.

Prototype vs build: testing a flow means building a prototype sufficient to run it against its required performance bounds — not a pilot-ready implementation. For flows with low fabrication cost, prototype and production version are the same thing; go straight to fabrication and skip the prototype step. The Minimum Representative Pilot (MRP) is the first point a customer touches a fully-fabricated system. Everything before the MRP is either prototype testing or individual flow fabrication. A customer at the MRP should feel the product is entirely real — because it is.

Section 5

The efficacy requirement

Every flow specification that involves a Working Product, Communications Product, or Using workflow must include an efficacy requirement. Efficacy is the property that distinguishes a flow that works from a flow that runs. A flow can execute without producing the state change it is designed to produce. Efficacy closes that gap.

Generic efficacy requirement — all flows

The flow must reliably produce the specified state change in its target actor, through the delivery mechanism and carrier defined in the ARM, without requiring performance above what the ARM assumes for that carrier.

Working Product flow efficacy requirement

The Working Product flow must eliminate the Key Monetizable Cost for any customer who completes the use routine, through the R2 mechanism, within the delivery mechanism and cost structure the ARM specifies.

The test is deliberately constrained: does the flow work when delivered by an average person in the specified role at the specified cost? Not an exceptional person. Not above-budget. Average, at spec. This test distinguishes an architectural failure from an execution failure.

If the flow fails the efficacy test because… It is a… Response
It only works when carried by an exceptional person Architectural failure ARM does not assume exceptional people. Return to ARM and Carrier design
It only works at above-spec cost Architectural failure FMOS is wrong. Return to FinSim with correct cost inputs
It works at spec but underperforms in a specific instance Execution failure Training, supervision, quality review — not architectural redesign
The state change is wrong for the target actor Architectural failure CTM is wrong. Return to the customer state change that the flow is designed to produce

The CDR gate

CDR is passed when all five outputs are complete and each meets its gate criterion. The gate is binary: pass or fail. A partial CDR package does not pass — it defers risk into the build phase, where it is more expensive to discover.

Flow register

Output 1
Flat list of all system elements with type, carrier, and source.
Gate criterion: complete — no gaps vs AOM cost lines and CTM product flows. No entries without AOM or CTM source.

Flow specifications

Output 2
One spec per flow — 11 fields including Carrier.
Gate criterion: all functions stated as measurable claims. All acceptance criteria specific enough to design a verification test from. Carrier named for every flow.

Interface Control Documents (ICDs)

Output 3
One per interacting flow pair.
Gate criterion: all interfaces defined. No flow sends data without an ICD. Failure modes named for every interface.

Requirements Traceability Matrix (RTM)

Output 4
One row per BALM requirement.
Gate criterion: all requirements traced. No unallocated flows. No rows without a named verification method. All 10 BALM requirements present.

Build sequence

Output 5
Numbered order with rationale, T4 assumption embedded in each flow, and FAIL consequence.
Gate criterion: risk-ordered (not dependency-ordered). Every T4 flow's prototype test defined — what is being tested, against what bound, with what FAIL consequence. Every T4 FAIL consequence is an architectural return, not an execution fix.
⚠ CDR failure modes

Primary: specs that describe what a flow is rather than what it must do, and acceptance criteria that cannot be turned into tests. A spec that says "the ITP referral protocol will allow ITPs to refer claimants" has described a feature. A spec that says "completable by an ITP staff member in ≤5 minutes per claim without legal training" has set a testable standard.

Secondary: build sequences ordered by dependency but not by risk — designing and prototyping T2 flows before testing T4 ones and leaving the hardest bets until the end. The sequence should be: Part1 (T4) → CP3 (T4) → WP2 (T4) → CP1/WP1 (T4) → Part2 (T3) → ... not the reverse.

Tertiary: Carrier field left blank or set to "TBD." Carrier determines the design specification. A blank Carrier means the spec is not yet a spec — it is a placeholder.
